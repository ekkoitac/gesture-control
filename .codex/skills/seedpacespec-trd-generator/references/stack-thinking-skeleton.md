# 技术栈思考骨架 — 无专属 Reference 时的 §4 展开指引

> 当技术栈的 `loaders.yaml.conditions` 为空或全部不命中时，本文件作为 §4 展开的思考框架。
> 这不是模板——不需要每层每问题都写。根据当前 R 的实际涉及范围选择相关层展开，
> 但展开深度必须对标黄金样本（设计意图 ≥3 句话、核心决策 ≥5 句话、伪代码+解读）。

---

## 后端/服务端

### §4 各层的核心设计问题

| 层 | 必须回答的问题 | 典型产出 |
|---|---|---|
| **接口层**（Controller/Handler/Resolver） | 路由设计、请求校验策略、响应格式标准化、错误码体系、版本策略、幂等设计 | 接口签名表（方法+路径+入参+出参+错误码）、请求校验规则、限流/熔断策略 |
| **服务层**（Service/UseCase） | 业务编排逻辑、事务边界在哪里、跨服务调用如何处理失败、领域规则校验 | 编排伪代码+设计解读、事务范围标注、补偿/回滚策略、领域校验规则表 |
| **领域层**（Domain/Entity/VO） | 实体建模、聚合边界、不变量约束、领域事件 | 实体关系图、聚合根识别、不变量清单、事件定义 |
| **数据层**（Repository/DAO/ORM） | Schema 设计、索引策略、查询模式（读写分离?）、数据迁移方案 | 表结构定义、索引设计依据、迁移脚本要点、缓存策略 |
| **基础设施层**（MQ/Cache/外部服务） | 消息投递保证（at-least-once?）、缓存一致性、外部服务降级策略 | 消息格式、消费幂等设计、缓存更新策略、降级兜底方案 |

### 设计意图"必答题"

每个核心模块的设计意图至少覆盖以下维度（按实际涉及选择，不强求全写）：

**并发安全**
- 这段逻辑是否会被并发调用？
- 锁策略：悲观锁/乐观锁/无锁？粒度（行级/表级/分布式）？
- 并发写同一资源时的冲突解决（last-write-wins / CAS / 排队）

**事务边界**
- 哪些操作必须原子成功或原子失败？
- 单库事务还是跨库/跨服务？
- 跨服务一致性：saga / TCC / 最终一致？补偿逻辑是什么？

**幂等性**
- 同一请求重复到达会怎样？
- 幂等键设计（请求 ID / 业务唯一键）
- 幂等窗口（多久内算重复）

**失败恢复**
- 依赖方（DB/缓存/下游服务）不可用时的行为？
- 重试策略：指数退避？最大次数？哪些错误可重试？
- 降级策略：返回缓存/默认值/部分结果？
- 熔断：触发条件和恢复策略？

**可观测性**
- 关键路径的日志级别和内容（入口/出口/异常/慢查询）
- 核心指标（QPS/延迟/错误率/队列深度）
- 链路追踪：trace 贯穿哪些服务？span 粒度？

**数据一致性**
- 缓存与 DB 的一致性策略（Cache-Aside / 写穿透 / 事件驱动刷新）
- 多副本/多数据源之间的同步机制
- 脏读/幻读是否可接受？隔离级别选择依据

## 嵌入式

### §4 各层的核心设计问题

| 层 | 必须回答的问题 | 典型产出 |
|---|---|---|
| **应用层**（Task/Thread/State Machine） | 任务划分依据、优先级分配、状态机设计、任务间通信方式 | 任务表（名称+优先级+栈大小+触发条件）、状态机图、消息/事件定义 |
| **服务/中间件层**（Protocol/Codec/Manager） | 协议解析策略、缓冲区管理、超时处理、重传机制 | 协议帧格式、解析状态机、缓冲区大小依据、错误恢复流程 |
| **驱动层**（Driver/HAL/BSP） | 硬件初始化序列、中断处理策略（ISR 里做什么/不做什么）、DMA 配置 | 初始化时序图、ISR 处理清单（耗时约束）、寄存器配置表 |
| **平台抽象层**（OS Abstraction/Board Config） | 可移植性边界、平台差异封装、条件编译策略 | 抽象接口定义、平台适配点清单 |

### 设计意图"必答题"

**实时性约束**
- 这段代码的最大执行时间（WCET）？
- 会不会阻塞高优先级任务？
- ISR 中只做标记（set flag/send event）还是做实际处理？
- 如果需要长时间处理，如何分拆到任务级别（deferred processing）？

**内存预算**
- 本模块的 RAM 占用（static + stack + heap）？
- 是静态分配还是动态分配？动态分配的上界是多少？
- 栈大小如何确定（最深调用链分析）？是否有栈溢出检测？
- Flash 占用对 OTA 分区的影响？

**资源竞争**
- 中断和主循环/低优先级任务是否共享数据？
- 保护机制：关中断 / 互斥锁 / 原子操作 / 无锁队列？
- 临界区范围（越小越好——标注最大持有时间）
- 优先级反转风险：是否需要优先级继承？

**功耗影响**
- 本模块活跃时的功耗等级？
- 空闲时能否进入低功耗模式？如何唤醒？
- 外设使用完毕是否需要关闭时钟/电源？
- 对整机续航的量化影响估算

**硬件约束**
- 时钟频率/带宽限制如何影响设计？
- 引脚复用冲突？
- 外设的最大传输速率/FIFO 深度/中断响应延迟？
- 温度/电压范围对设计的影响？

**故障处理**
- 硬件异常（通信超时/CRC 校验失败/外设无响应）如何检测？
- 恢复策略：重试/复位外设/上报错误/触发看门狗？
- 看门狗策略：喂狗时机、超时配置、复位后的恢复逻辑
- 错误累积计数与降级策略（连续 N 次失败后放弃）

## 前端/Web/跨端

### §4 各层的核心设计问题

| 层 | 必须回答的问题 | 典型产出 |
|---|---|---|
| **页面/路由层**（Page/Screen/Route） | 路由结构、页面入口数据获取方式、页面级状态作用域、布局策略 | 路由表、页面数据依赖图、SSR/CSR 选择依据 |
| **容器/业务组件层**（Container/Feature Component） | 数据来源（props/store/fetch）、内部状态划分、子组件编排、异步流程处理 | 组件树（标注数据流向）、状态归属表、异步时序图 |
| **展示组件层**（UI/Presentational） | props 契约、样式策略、响应式/自适应方案、无障碍（a11y） | Props 接口定义、变体枚举、样式方案选择依据 |
| **逻辑层**（Hook/Service/Utils） | 可复用逻辑抽取边界、输入输出契约、缓存/去抖/节流策略、副作用隔离 | Hook/Service 接口签名、消费者清单、边界情况清单 |
| **状态管理层**（Store/State/Context） | 状态划分粒度、持久化策略、跨组件通信方式、乐观更新 | Store slice 定义、action/reducer/selector 设计、状态流转图 |
| **数据请求层**（API/Repository/DataSource） | 请求封装模式、缓存策略（SWR/React Query 等）、错误统一处理、类型映射 | API 函数签名、请求/响应类型、错误处理策略表 |

### 设计意图"必答题"

**组件拆分决策**
- 这个 UI 块为什么是独立组件而不是父组件的一部分？拆分的驱动力是复用/职责单一/性能？
- 受控 vs 非受控？状态提升到哪一层？为什么？

**状态归属**
- 这份状态属于组件本地、共享 store、还是 URL？判定依据？
- 派生状态 vs 存储状态？是否有冗余？
- 状态变更的触发链路（用户操作 → action → 哪些 UI 更新）

**异步流程**
- 数据加载的时机和策略（进入页面/滚动到可见/用户触发）
- loading/error/empty 三态如何处理？骨架屏/重试/兜底 UI？
- 竞态处理（快速切换页面/连续请求取消旧的）
- 乐观更新 vs 等待确认？回滚策略？

**性能**
- 哪些组件需要避免不必要的重渲染？memo/useMemo/useCallback 的使用依据？
- 大列表虚拟化？图片懒加载？代码分割边界？
- 首屏关键路径上有哪些阻塞资源？

**复用与抽象**
- 提取 Hook/工具函数的时机和边界（两次使用才抽取？还是预判复用？）
- 组件的泛化程度（业务绑定 vs 通用 UI 组件）

## Android/移动端

### §4 各层的核心设计问题

| 层 | 必须回答的问题 | 典型产出 |
|---|---|---|
| **UI 层**（Activity/Fragment/Compose Screen） | 页面生命周期管理、UI 状态恢复（config change/进程回收）、导航设计 | 页面清单、Navigation Graph、状态保存策略 |
| **ViewModel/Presenter 层** | UI 状态建模（UiState）、事件处理（单次事件 vs 持续状态）、业务逻辑编排 | UiState 数据类、Event/Effect 定义、状态流转图 |
| **Domain/UseCase 层** | 业务规则封装、跨 Repository 编排、线程/协程调度策略 | UseCase 接口签名、编排伪代码、调度器选择依据 |
| **Data/Repository 层** | 数据来源策略（网络优先/缓存优先/离线优先）、多数据源合并、分页 | Repository 接口、缓存策略表、数据同步机制 |
| **基础设施层**（Network/DB/Platform） | Retrofit/Room/DataStore 配置、序列化、拦截器链、平台 API 适配 | 网络配置、数据库 Schema、拦截器清单 |

### 设计意图"必答题"

**生命周期与状态恢复**
- Activity/Fragment 被系统回收后，哪些状态需要恢复？SavedStateHandle vs ViewModel vs 持久化？
- Configuration change（旋转/多窗口）时的行为？
- 后台进程被杀后的冷启动恢复路径？

**线程与协程**
- 哪些操作必须在主线程？哪些必须离开主线程？
- 协程作用域选择（viewModelScope/lifecycleScope/自定义）—— 取消时机？
- 并发请求策略（并行/串行/互斥）？取消旧请求？

**内存与性能**
- 大图/长列表的内存策略（Glide 缓存/RecyclerView 复用/Compose LazyList）
- 内存泄漏风险点（Context 引用/注册未反注册/协程未取消）
- ANR 风险（主线程阻塞场景识别）

**多模块/组件化**
- 模块间通信方式（路由/接口下沉/事件总线）
- 模块依赖方向（禁止反向依赖的约束如何保证）
- 公共资源归属（字符串/图片/主题/基础组件）

**平台差异与兼容**
- minSdk 约束下的 API 可用性？需要 compat 库的场景？
- 厂商定制 ROM 的兼容问题（通知/后台限制/权限）
- 多屏幕适配策略（dp/sp/自适应布局/折叠屏）

**数据持久化与同步**
- 本地缓存过期策略？
- 离线操作队列（先写本地，网络恢复后同步）？冲突解决？
- 敏感数据加密存储方式？

## 使用原则

1. **选择性展开**：根据当前 R 的实际涉及范围选层和选问题，不需要填满每一格
2. **深度不打折**：没有专属 reference ≠ 可以写浅。Full 级别的每层设计意图仍须 ≥3 句话（核心决策 ≥5 句话）
3. **伪代码+解读不可省**：关键编排逻辑必须有伪代码块 + 中文设计解读（阐明步骤依赖和兜底策略）
4. **"必答题"是思考工具**：在写设计意图时逐条过一遍，命中的展开写，不命中的跳过——确保不遗漏该技术栈特有的设计维度
