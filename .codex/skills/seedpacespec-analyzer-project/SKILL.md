# seedpacespec-analyzer-project

## Goal

对已 clone 到本地的代码仓库进行结构化洞察，生成 `architecture.md` 全局上下文文档。

**定位**：有深度的客观事实性前置上下文——不做主观判断、不推断动机、不替下游决策。

「有深度」= 不搬运单文件可读的信息，而是做**关联分析**——将分散在多个文件中的客观事实交叉关联，形成单独读某个文件无法获得的全局视图。

**写作原则**：

- **不搬运**：Agent 读单个文件就能获取的信息不重复罗列，改为指向文件路径
- **要关联**：将分散在多处的客观事实交叉关联（如大文件 × 高频变更 × 无测试 = 交叉风险信号）
- **要标注不一致**：声明但未使用的依赖、同功能多实现、命名不一致等客观矛盾必须记录
- **不推测**：不推断"为什么选择了 X 而不是 Y"，只记录"项目中同时存在 X 和 Y"

**不做的事**：逐行代码分析、缺陷判断或修复建议、操作指南或开发模板、解析构建产物/生成代码/第三方依赖源码、推断设计动机或选型原因。

### 下游用途（须在生成的 architecture.md 中呈现）

生成文档时**必须**按 `templates/architecture-output.md` 写入 **「文档用途（下游消费指引）」**整节（置于 §0 之前），内容与表格以模板为准，不得省略。

| 分析产出 | 下游侧重 | 摘要 |
|----------|----------|------|
| **§4 核心业务域** | 迭代类、优化类需求；与 PRD、`global-specs/changelog.md` 对照 | 把「历史上做过的业务能力」落到域与路径；新 PRD 来时做语义对照——例如 §4 已写明「家长音视频通话」相关域与边界，新需求仍是同一主题时，大模型/Agent 可识别与既有域重合，判断域内演进 vs 新辟模块。 |
| **§3 / §2、分层映射** | `design.md`、TRD、方案设计 | 让分层、挂载点、依赖边界与仓库现状一致，方案可落地。 |
| **§9 风险热点** | `sdx-detect`、缺陷检测 | 耦合、大文件、高频变更、交叉矩阵——收窄质检与改动风险关注面。 |
| **§6 编码约定** | `sdx-apply`、生成代码与 Review | 提取并沿用本项目命名与工程规范。 |

## State Machine Gates

⛔ 约束: 任何门禁未完成 → 只能向用户确认/请求输入，不得扫描仓库或写入文件。

| #  | 状态              | 完成条件                                         | 未完成时行为         |
|----|-------------------|--------------------------------------------------|----------------------|
| G1 | `mode_selected`   | 用户选定 A/B/C/D（见阶段 0）                    | 向用户询问模式 |
| G2 | `domains_confirmed` | 全量模式下：业务域候选列表已经用户确认（见阶段 3）| 向用户确认域   |
| G3 | `executing_or_done` | 扫描完成 / 增量更新完成 / 跳过                   | 按选择执行           |

## 前置条件

### G1 门禁交互

⛔ 约束: G1 未完成前，禁止扫描仓库或写入任何文件。

**无论 `architecture.md` 是否存在**，都必须使用 Codex 用户确认（单选）让用户选择模式：

| 选项                      | 行为                                                           |
|---------------------------|----------------------------------------------------------------|
| **A. 全量生成**           | 执行阶段 1-5，覆盖写入根目录 `architecture.md`                |
| **B. 增量更新**           | 基于 commit diff，仅更新受影响章节                             |
| **C. 跳过，使用已有文档** | 不改写文件，直接结束                                           |
| **D. 跳过，不生成**       | 不做任何操作，直接结束                                         |

**选项可用性规则**：

- `architecture.md` 已存在 → A、B、C、D 全部可选
- `architecture.md` 不存在 → 仅展示 A 和 D

**各选项后续流程**：

- **A** → 执行阶段 1-5，覆盖写入
- **B** →
  1. 提示用户输入 `commit_range`（可选），格式：`abc123..def456`
  2. 若未提供，从 `architecture.md` 头部 `基准提交` 字段提取 old_commit，以 `old_commit..HEAD` 推导
  3. ⛔ 约束: 若 `基准提交` 字段缺失/格式无效/commit hash 不存在 → 必须向用户确认/请求输入：「无法自动推导 commit range（原因：<具体原因>），请手动输入或选择全量生成」
  4. 进入增量模式流程
- **C** → ⛔ 约束: 禁止覆盖、删除或截断 `architecture.md`。输出确认后结束。**流程终止。**
- **D** → 输出确认后结束，不创建任何文件。**流程终止。**

### 执行模式

- **全量模式**：`architecture.md` 不存在或用户选择全量生成 → 执行阶段 1-5
- **增量模式**：提供 `commit_range` 且 `architecture.md` 已存在 → 仅更新受影响章节：
  - 变更涉及包管理文件 → 更新技术栈（§2）
  - 变更涉及目录结构调整 → 更新架构设计（§3）
  - 变更涉及业务代码 → 更新对应业务域（§4）+ 风险热点（§9）
  - 变更涉及公共模块 → 更新风险热点（§9）
  - 在头部追加增量更新摘要（含 commit range 和变更概述），保留未变更章节

### 输入参数（自动获取）

| 参数           | 来源                                                                                     |
|----------------|------------------------------------------------------------------------------------------|
| `repo_path`    | 当前工作区根目录（自动读取）                                                             |
| `focus_paths`  | 默认为空（全仓库）；monorepo 时 **必须向用户确认/请求输入** 让用户选择聚焦范围，禁止自动推断    |
| `exclude_dirs` | 自动排除：`build`, `node_modules`, `.dart_tool`, `dist`, `__pycache__`, `.idea`, `android/build`, `ios/Pods`, `vendor`, `target` |
| `commit_range` | 增量模式：用户提供或从 `architecture.md` 头部基准 commit 推导；全量模式忽略               |

## 执行流程

### 阶段 1：前置探测

- □ 仓库根目录是否存在
- □ 识别项目类型：package.json / pubspec.yaml / requirements.txt / Cargo.toml / go.mod / pom.xml / build.gradle 等
- □ 识别项目形态：application / sdk / hybrid
  判定信号：
  - package.json 有 main/exports/types/typings 字段且无 pages/screens/app 目录 → sdk
  - build.gradle 含 com.android.library → sdk
  - Cargo.toml 含 [lib] 且无 [[bin]] → sdk
  - setup.py/pyproject.toml 有 packages= 且无 manage.py/app.py → sdk
  - go.mod 存在但无 main.go 或 cmd/ → sdk
  - 以上均不满足 → application
  - SDK 信号命中但同时有 example/demo/playground 目录 → hybrid

  ⛔ 约束: 必须向用户确认/请求输入 让用户确认项目形态（自动检测可能误判）：
  > 基于代码信号，本项目的形态判定为「{检测结果}」。请确认：
  > - application（业务应用——有页面/入口/用户交互）
  > - sdk（SDK/库/工具包——对外暴露 API，被其他项目消费）
  > - hybrid（SDK + 示例应用——SDK 为主体，附带 demo/playground）

  用户确认后，将结果写入 architecture.md 的 Agent 速览「项目形态」字段。

- □ 是否是 monorepo（多个包管理文件、workspace 配置）
- □ 统计仓库规模：文件数、代码行数、语言分布
- □ 读取 .gitignore 排除无关目录
- □ 获取当前 HEAD commit hash（用于后续增量更新基准）

---

### 阶段 2：技术栈与架构识别

**技术栈识别**：

- 读取包管理文件，提取所有直接依赖及版本
- 按类别归类：语言/SDK、框架核心、状态管理、路由、网络、存储/ORM、UI、测试、构建工具
- 标注客观事实性异常：
  · 声明但未在源码中检出 import/require/use 的依赖（逐一检查）
  · 同一功能存在多种实现方式（如同时使用 fetch 和 axios）
  · 依赖版本间的已知不兼容
  · 命名不一致（如目录/文件拼写错误）
- 构建配置：构建工具、产物路径、环境配置
- 开发脚本：启动/构建/测试/lint 命令

**架构分层识别**（从项目实际目录结构中归纳，不限于特定框架）：

1. 从项目实际目录和代码引用关系中**归纳**出分层结构。层名、层数、层级关系完全由项目决定，不套用固定模板。以下仅为常见分层的**思考参考**（不同技术栈差异很大）：

    - 前端典型：UI/视图层、逻辑/Hook 层、状态管理层、模型/类型层、数据请求层、基础设施层
    - 后端典型：Controller/Handler 层、Service 层、Domain/Entity 层、Repository/DAO 层、Infrastructure 层
    - 移动端典型：Screen/Widget 层、Bloc/ViewModel 层、Model 层、Repository 层、Platform 层

    ⛔ 约束: 一切以代码中的实际目录结构和调用关系为准，禁止套用固定模板。

2. 识别入口点：
    - 应用入口文件
    - 路由/API 注册配置
    - 多入口架构（多个 App 壳、微服务入口、微前端入口）

3. **输出分层架构视图**（下游 TRD/design 的核心消费项）：
   - 用架构分层思维对项目目录做语义归类，输出"层名→目录→职责→层角色"映射表
   - 明确层间调用规则：哪层可以调用哪层、禁止反向调用的约束
   - 标注项目实际违反分层规则的地方（如 UI 组件直接调了 API、store 直接操作 DOM）
   - 每层给一句话"层角色"描述，让 TRD 方案章节可直接引用

---

### 阶段 3：业务域识别与用户确认（→ G2 门禁）

1. 通过以下客观信号自动识别候选业务域：
    - 路由/API 端点定义中的独立路由组（每组 = 候选域）
    - 数据模型/实体定义（每个核心 Model/Entity 对应的功能模块 = 候选域）
    - 独立的状态管理/服务单元（每个独立 store slice / service / bloc = 候选域信号）
    - 目录结构中的语义分组（如 user/, order/, payment/, auth/）

2. 将识别结果以列表形式展示给用户，**使用 向用户确认/请求输入** 让用户：
    - 确认/调整业务域列表（可增删）
    - 标记哪些是核心域（需深度分析）、哪些是辅助域（简略分析）

3. 用户确认后 G2 门禁通过，进入阶段 4

⛔ 约束: 禁止跳过用户确认直接分析。禁止仅按文件数/行数判断核心域。

---

### 阶段 4：深度分析

**业务域分析**（按用户确认的分类执行不同深度）：

【核心域 — 深度分析】：

- 域名称、路径、规模（文件数、总行数）
- 内部结构：域内各子目录/子模块的文件数和职责（使用项目实际目录命名）
- 域内数据流路径：从入口到出口的实际调用链，标注每一跳经过的文件
- 域边界：
  · 对外暴露：被其他域/模块引用的导出
  · 外部依赖：引用的其他域和公共模块
- 域内复杂度信号：最大单文件行数、最多分支逻辑的文件
- 关键抽象：识别该域中的核心抽象（类、接口、模块），不限于特定框架
- **领域模型提取**（仅当代码中**客观存在**可归纳材料时撰写；从 types/interfaces、model/entities、reducer state shape、action payload、API 响应结构等提取）：
  · 实体清单：列出域内可验证的业务实体，标注类型定义所在文件、关键字段及字段语义
  · 实体关系：实体之间的组合、聚合、引用、继承关系
  · 状态枚举与流转：域内的状态常量、枚举值及其含义、状态流转条件
  · ⛔ 约束: 若无独立实体层/无稳定类型模型/无可归因的业务对象边界 → 禁止编造或套空表；用一两句客观话说明即可，整节省略
- **业务协同与编排**（同样仅基于事实；无则省略）：
  · 核心业务场景识别：该域承载哪些关键业务流程
  · 实体参与图：每个核心场景中参与的实体、各自职责、交互顺序
  · 编排机制：谁驱动实体间的协同，标注编排代码所在文件
  · ⛔ 约束: 若无可识别的多步编排或实体参与链 → 不写空场景模板；可省略或合并为一句说明

【辅助域 — 简略分析】：

- 域名称、路径、规模、一句话职责

**跨域协作路径**（记录客观事实，不做评价）：

- 跨域端到端数据流路径：贯穿多个域的完整调用链，标注每一跳经过的文件

**编码约定提取**（记录已形成的事实约定）：

- 命名规范（文件、类/模块、函数、变量的实际命名模式）
- 标注不一致：不符合主流命名模式的文件或目录
- 状态管理使用方式
- API/请求调用链路
- 路由注册和参数传递方式
- 国际化方式
- lint 规则（已启用/已禁用的关键规则）

**风险热点采集**（客观记录，不做严重性判断）：

- 大文件清单：超过 500 行的文件（文件名、行数、所属域）
- 耦合热点：被引用次数 TOP 15 的文件（含引用方列表）
- 平台/环境差异分支分布
- 高频变更文件（git log）：近 50 次提交中修改次数 TOP 10
- 无测试保护的核心模块清单
- 【交叉风险矩阵】：将大文件、高频变更、无测试、高被引用四项信号交叉关联，输出综合表

**测试与安全现状**：

- 测试文件位置、命名规则、框架
- 各模块的测试覆盖程度（有/无/部分）
- 密钥管理、认证鉴权、输入验证、敏感数据存储方式

## Guardrails

⛔ 以下为全局红线，部分已在执行流程中内联重申。

- ⛔ 状态机门禁（G1→G2→G3）是 ASSERTION：G1 未完成 → 禁止扫描或写入。G2 未完成 → 禁止进入业务域深度分析。凡需要用户交互的步骤，必须停下等用户明确回复，禁止跳过、合并或替用户做决定。
- ⛔ 技术栈无关：分层名称、关键抽象、分析维度均从项目实际代码中归纳，不硬编码特定框架术语。
- ⛔ 业务域判定：通过路由/API 端点、数据模型/实体、状态管理单元、目录语义等客观信号识别候选域，必须经用户确认。禁止仅按文件数/行数排序来判定核心域。
- ⛔ 核心域 ≠ 必有领域模型：用户将某域标为核心域只表示分析粒度与优先级，不表示必须写出实体清单。若客观上无实体/无稳定领域模型/无多步业务编排，诚实省略，禁止套模板填空或虚构实体。
- ⛔ 依赖分析不可跳过：输出结构（`templates/architecture-output.md`）§9.2 耦合热点中的公共模块被引用 TOP 10 和跨域依赖清单为必须完成项，禁止出现"本稿不展开"等跳过说明。
- ⛔ 交叉风险矩阵：输出结构 §9.5 必须将大文件、高频变更、无测试、高被引用四项信号交叉关联，只列命中数不做主观判断。
- ⛔ 输出结构 §5 职责边界：§5 只记录跨域的端到端数据流路径。域内数据流由 §4 各核心域章节负责，接口契约格式由 §6 编码约定负责，禁止重复。
- ⛔ 不推测：禁止推断设计动机、选型原因。只记录客观可验证的代码事实。
- ⛔ monorepo：必须向用户确认/请求输入 让用户选择聚焦的包/模块，禁止自动推断 core 包。
- ⛔ 超大文件（>1000 行）：仅记录文件名和行数，不做内容分析。
- ⛔ 覆盖写入仓库根目录（增量模式下保留未变更章节），每次附上基准 commit hash。
- ⛔ 生成 `architecture.md` 时**不得省略**模板中的「文档用途（下游消费指引）」章节（导读与 §0～§9 客观事实并存）。
