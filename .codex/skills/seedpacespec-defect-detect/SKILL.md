# seedpacespec-defect-detect



## Goal

对 seedpacespec 变更执行**业务缺陷检测流水线**，输出分级缺陷报告。

**流水线**：收集输入 → architecture.md 增量更新 → PRD 结构化 → TRD 结构化 → Git 变更画像 → 缺陷分析 → 缺陷报告。

**核心原则**：
- **确凿证据原则**：无证据不判定缺陷；不确定时用搜索工具获取真实代码再判断，禁止猜测
- **逐行审查**：不要发现一个 bug 就停止，审查所有变更代码
- **职责边界**：后端接口正确性属后端职责不判定；非本变更引入的历史问题不判定
- **不写业务代码**：只输出报告



## State Machine Gates

执行前必须依次通过以下状态门禁。**任何状态未完成 → 只能向用户确认/请求输入，不得跳入后续步骤。**

| # | 状态 | 完成条件 | 未完成时行为 |
|---|------|---------|-------------|
| G1 | `inputs_collected` | 变更已选定、PRD/TRD 路径已确认、Git 比较范围已确定 | 向用户确认/请求输入 收集缺失信息 |
| G2 | `architecture_updated` | architecture.md 已增量更新（或已是最新） | 执行增量更新 |
| G3 | `prd_extracted` | PRD → 需求 JSON 已在内存中 | 执行提取 |
| G4 | `trd_extracted` | TRD → 技术规格 JSON 已在内存中（已映射 PRD 业务规则） | 执行提取 |
| G5 | `changes_traced` | Git 变更画像已生成（含函数级 diff、trace_hints） | 执行变更分析 |
| G6 | `defects_reported` | 缺陷报告已生成并输出 | 执行缺陷检测 |

**状态推进单向** G1→G2→…→G6，不得跳跃。G2~G6 为自动串行，无需用户逐步确认。但 **G1 完成后、G2 开始前**，必须展示即将执行的流水线参数摘要并用 **Codex 用户确认/输入流程** 确认：「即将执行缺陷检测流水线，参数如下：\n- 变更：`<name>`\n- PRD：`<path>`\n- TRD：`<path>`\n- Git 范围：`<source>...<target>`（若 target 为 WORKTREE 则显示 `<source>...工作区（含未提交）`）\n- 聚焦模块：`<path 或 无>`\n\n确认开始？」


**🚫 禁止编造跳过理由**：AI 不得以任何自创名义（如恢复协议、追加模式、精简流程、复用已有产物、简化模式等）跳过门禁。所有门禁必须逐一通过并产出可验证输出。如果你发现自己正在向用户提议跳过某个门禁——立即停止并按正常流程执行。

## 执行流程

### 步骤 1: 收集输入（→ G1）

缺失时用 **Codex 用户确认/输入流程** 收集：

| 信息 | 来源 | 必需 |
|------|------|------|
| **变更名** | 用户提供 或 `seedpacespec list --json` 选择 | ✅ |
| **PRD 路径** | 变更目录下 `specs/` 或 `*_prd.md` | ✅（自动探测，找不到则问） |
| **TRD 路径** | 变更目录下 `*_trd.md` 或 `trds/` | ✅（自动探测，找不到则问） |
| **Git 比较基准** | 用户提供（如 `main`、`HEAD~5`、commit hash） | ✅ |
| **Git 比较目标** | 默认 `HEAD`；可选 `WORKTREE`（含未提交的暂存区 + 工作区修改） | 否 |
| **聚焦模块** | 用户可指定某个目录优先分析；若用户未指定，默认分析变更涉及的全部路径（不自行缩小范围） | 否 |

**自动探测路径逻辑**：
```
变更目录 = seedpacespec/changes/active-change/<变更名>/
PRD: specs/*-prd.md → *_prd.md → *.prd.md → 向用户确认/请求输入
TRD: trds/*.md → *_trd.md → 向用户确认/请求输入
```

**Git 比较范围选择**：

用户未提供时：

1. 运行 `git log --oneline -20` 获取最近 20 条 commit
2. 运行 `git branch -a --sort=-committerdate | head -10` 获取活跃分支
3. 用 **Codex 用户确认/输入流程** 以列表形式让用户**分别**选择 source 和 target：

**第一步：选择比较基准（source）**

```
请选择 Git 比较基准（起点）：

分支：
  A) main
  B) develop
  C) feature/xxx

最近 Commits：
  D) abc1234 - feat: 添加年龄切换功能 (2天前)
  E) def5678 - fix: 修复卡片详情页崩溃 (3天前)
  F) ghi9012 - refactor: 重构数据层 (5天前)
  ...

  Z) 手动输入 commit hash 或分支名
```

**第二步：选择比较目标（target）**

```
请选择 Git 比较目标（终点）：

  A) HEAD（默认，当前最新提交）
  B) 工作区（含未提交修改）— 检测暂存区 + 工作区的未提交代码
  C) abc1234 - feat: 添加年龄切换功能 (2天前)
  D) def5678 - fix: 修复卡片详情页崩溃 (3天前)
  ...

  Z) 手动输入 commit hash 或分支名
```

用户选择后确定比较范围：
- 选 **HEAD** 或具体 commit → `source..target`（commit 间比较）
- 选 **工作区** → `target` 设为 `WORKTREE`，后续 Step 5 使用 `git diff {source}` 比较 source commit 与当前工作区（含暂存区 + 未暂存修改）

若用户在第二步直接确认，默认 target = `HEAD`。

### 步骤 3: PRD 结构化提取（→ G3）

将 PRD Markdown 转为**机器可读的 JSON 数组**，供后续步骤消费。

### 3.1 提取规则

- **引用而非复写**：JSON 中的描述来自 PRD 原文，不改写、不展开
- **一条需求一个对象**：与 PRD 中的 R-XXX 一一对应
- **只新增机器所需元数据**（如 confidence 枚举），不重复 PRD 已有的散文

### 3.2 提取流程

1. 读取 G1 确定的 PRD 文件
2. 对 PRD「需求清单」表格中的**每一行**，提取为一个 JSON 对象
3. 产出保存在**内存**中，不写入磁盘

### 3.3 输出格式

```json
[
  {
    "id": "R-001",
    "story": "PRD 中「用户故事」列的原文",
    "acceptance": ["验收标准 1（原文）", "验收标准 2（原文）"],
    "priority": "P0",
    "confidence": "confirmed"
  }
]
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 与 PRD 需求 ID 一致（R-XXX） |
| `story` | ✅ | PRD「用户故事」列原文 |
| `acceptance` | ✅ | PRD「验收标准」列，拆为数组 |
| `priority` | ✅ | P0 / P1 / P2 |
| `confidence` | ✅ | `confirmed`（PRD 明确）/ `inferred`（合理推断）/ `needs_confirmation`（需产品确认） |

**confidence 映射**：`[已确认]` → confirmed、`[推断]` → inferred、`[需确认]` → needs_confirmation。**无显式标记时默认 `needs_confirmation`**（不得自行升级为 `inferred`）

**约束**：
- 不要在 JSON 中添加 PRD 没有的信息
- `needs_confirmation` 的需求不作为后续缺陷检测的"定罪依据"

`✅ [2/5] PRD JSON 已提取（N 条需求：P0 x | P1 y | P2 z）`

### 步骤 4: TRD 结构化提取（→ G4）

从 TRD 文档中提取技术规格，**严格关联 PRD 业务规则与 TRD 技术实现**，输出结构化 JSON。

### 4.1 提取规则

- **业务 + 技术一一对应**：每条规格同时包含 PRD 的业务规则和 TRD 的技术实现
- **完整不省略**：即使规则与其他需求重复，也需完整复述，不得使用"同上""同 R-001"
- **精确引用**：模块名、接口名、类名、方法名、文件路径必须与 PRD/TRD 原文一致

### 4.2 提取流程

1. 使用 G3 产出的 PRD JSON
2. 读取 G1 确定的 TRD 文件
3. 对每条 PRD 需求（R-XXX）：
   - 查看 TRD「需求覆盖矩阵」（若有），定位该需求对应的 TRD 章节
   - 从对应章节中提取技术实现细节（文件、类、方法、改动描述）
   - 生成一条规格，同时包含 PRD 业务规则和技术实现
4. TRD 中无对应实现的需求 → 不输出该条规格，但**必须在报告的「通过的检查项」中标注**：`R-XXX | ⚠️ 未覆盖 | TRD 中无对应技术方案，建议人工核对`
5. 产出保存在**内存**中，不写入磁盘

### 4.3 specific_requirements 的 2 类信息（核心）

`specific_requirements` 字段必须覆盖，**不得有任何省略**：

| 类别 | 内容来源 | 要求 |
|------|---------|------|
| **PRD 业务规则** | PRD JSON 的 `story` + `acceptance` | 包含场景、时机、所有规则、约束 |
| **TRD 技术实现** | TRD 改动方案章节（若有） | 精确到类名/方法名/文件路径 |

### 4.4 禁止性约束（红线）

- ❌ **禁止遗漏业务上下文**：必须保留 PRD 中的场景、时机、所有规则、约束
- ❌ **禁止技术细节孤立**：技术实现需与业务规则强关联
- ❌ **禁止模糊表述**：模块、字段、接口名需与 PRD/TRD 完全一致
- ❌ **禁止类比/省略**：即使规则重复也需完整复述——**违反则该规格无效**

### 4.5 Few-Shot 示例

**Bad Case**（缺失业务上下文）：
```json
{
  "specification_id": "TRD-001",
  "related_requirement_id": "R-001",
  "module": "normal_card_detail.dart、game_card_detail.dart",
  "specific_requirements": "根据 ages.length > 1 条件渲染年龄标签"
}
```
问题：未提及业务角色（家长）、场景（详情页）、完整规则（分段>1 才显示）。

**Good Case**（业务 + 技术，闭环完整）：
```json
{
  "specification_id": "TRD-001",
  "related_requirement_id": "R-001",
  "module": "家长端 APP 故事卡详情页、normal_card_detail.dart、game_card_detail.dart、music_card_detail.dart、ai_create_card_detail.dart",
  "specific_requirements": "1. PRD 业务规则：家长在故事卡详情页查看时，当该故事卡年龄分段>1 时看到年龄切换入口；仅 1 个年龄分段时不显示。2. TRD 技术实现：4 个详情页的 build 方法中，根据 _data.ages.length > 1 条件渲染可点击的年龄标签（Positioned 在封面右上角），否则不渲染该 Widget。"
}
```

### 4.6 输出格式

```json
[
  {
    "specification_id": "TRD-XXX",
    "related_requirement_id": "R-XXX",
    "module": "业务角色/场景、技术模块1、技术模块2",
    "specific_requirements": "1. PRD 业务规则：{完整业务规则}；2. TRD 技术实现：{完整技术细节}"
  }
]
```

`✅ [3/5] TRD 规格已提取（N 条，覆盖 PRD x/y 条）`

### 步骤 5: Git 变更画像（→ G5）

产出"本次变更画像"。**只描述事实与风险线索，不判断缺陷。**

### 5.1 技术栈探测

从 architecture.md 读取：主语言/源码扩展名、生成文件模式、架构分层目录约定、状态管理方案、核心源码根路径等信息。
不存在时从包管理文件推断，标注 `"tech_detection": "inferred"`。

### 5.2 分析流程

1. **Git Diff**：
   - **target 为 commit/分支时**：`git diff {source}..{target} -- {src_ext}`
   - **target 为 WORKTREE 时**：`git diff {source} -- {src_ext}`（无 `..`，比较 source commit 与当前工作区，含暂存区 + 未暂存修改）
2. **文件级**：按架构分层分类（UI/状态/模型/数据/工具），统计增删行数
3. **函数级**：识别变更的类/函数/方法，记录 symbol 名、变更类型（ADDED/MODIFIED/DELETED）、diff hunk 引用、变更摘要。框架特有标注（build→渲染变更、initState/dispose→生命周期、setState/emit→状态刷新等）
4. **状态管理变更**：状态定义/事件/动作的新增修改、触发调用变化、受影响消费方（import 反查）
5. **噪音过滤**：import 排序、纯格式化、注释、生成文件、lock 文件 → 记录不分析

### 5.3 风险标注（risk_notes）

基于 §5.2 的分析产出，逐条匹配以下规则。**命中 → 写入 risk_notes；未命中 → 不标注。**

| 风险信号 | 判定依据（基于 §5.2 产出） | 命中条件 |
|---------|--------------------------|---------|
| 核心业务链路引入新条件分支 | §5.2-3 函数级分析 | MODIFIED 类型的 symbol 新增了 if/switch/guard/when 分支，且该 symbol 位于 TRD §4 标注的主流程调用链上（或被 PRD 核心场景直接消费） |
| 硬件通信/支付/认证/NFC/蓝牙相关变更 | §5.2-2 文件级分类 | 变更文件路径或 import 含 `payment`/`auth`/`ble`/`nfc`/`hardware`/`crypto`/`certificate` 等领域关键词 |
| 公共模块/基类/共享组件变更 | §5.2-2 文件级 + §5.2-4 import 反查 | 变更文件被 ≥3 个其他模块引用（含直接+间接），或位于项目约定的 `shared`/`common`/`base`/`utils` 目录 |
| 公共方法签名变更 | §5.2-3 函数级分析 | MODIFIED 的导出函数/方法：参数列表（数量/类型/顺序）或返回类型发生变化 |
| 数据模型字段变更 | §5.2-3 函数级分析 | Model/Entity/Type/Interface/Schema 定义中字段增删改（字段名/类型/可选性变化） |
| 同一需求多文件实现不一致 | §5.2-3 交叉比对 | 同一 R（normalizedKey）涉及的 ≥2 个文件中，对同一业务规则的实现逻辑存在矛盾（如一处校验了边界另一处未校验） |
| 阻塞/耗时操作出现在关键路径前 | §5.2-3 函数级分析 | 新增的同步 IO/网络请求/数据库查询/文件读写出现在 UI 渲染路径、事件处理主循环、或中断处理函数内 |
| 并发操作可能竞态 | §5.2-4 状态管理变更 | ≥2 个异步调用写同一状态字段且无互斥保护（无 lock/mutex/serial queue/单一写入点） |
| architecture.md 风险热点文件被修改 | §5.4 热点匹配 | 变更文件命中 architecture.md §9 任一热点清单（大文件/耦合热点/交叉风险矩阵） |

**规则优先级**：一个文件/函数可同时命中多条规则，全部写入 risk_notes（不去重、不互斥）。

### 5.4 architecture.md 风险热点关联（必做）

读取 architecture.md §9，将变更文件与以下热点信号交叉匹配：

| 热点信号 | 匹配方式 | 命中后的检测增强 |
|----------|----------|-----------------|
| **大文件（§9.1）** | 变更文件在大文件清单中 | Step 6 对该文件逐函数检查：修改是否越过函数边界、是否意外改动相邻逻辑 |
| **耦合热点（§9.2）** | 变更文件在耦合热点清单中 | Step 6 追踪所有引用方，检查调用兼容性（签名、语义、默认值是否 break） |
| **平台/环境差异（§9.3）** | 变更文件在平台差异分布中 | Step 6 检查修改是否只影响目标环境，确认其他环境的条件分支未被波及 |
| **交叉风险矩阵命中 ≥2（§9.5）** | 变更文件在矩阵中 | 将该文件的检测优先级提至最高，综合执行上述所有增强项 |

将匹配结果写入变更画像的 `hotspot_matches` 字段，传递给 Step 6。

### 5.5 检测线索（trace_hints）

给 Step 6 的具体建议，示例：
- "横向对比 4 个详情页的 handleXxx 是否一致"
- "新增条件 >= 1，PRD 要求 > 1"
- "BaseWidget 签名变更，反查子类是否适配"
- "await fetchXxx() 写在 readCard() 前，检查是否应后置"

`✅ [4/5] 变更画像已生成（N 个文件，+X/-Y 行，M 条检测线索）`

### 步骤 6: 业务缺陷检测（→ G6）

汇总 G2~G5 全部产出，结合代码**全链路上下文**检测缺陷。

**trace_hints 优先消费（必做）**：Step 5 产出的 `trace_hints` 是具体检测建议（如"横向对比 4 个详情页的 handleXxx 是否一致""新增条件 >= 1，PRD 要求 > 1"）。Step 6 开始前先逐条阅读 trace_hints，**每条 hint 必须在对应校验项中执行并在报告中标注结果**（确认无问题 或 发现缺陷）。trace_hints 不能仅出现在报告摘要中——它们是检测输入，不是输出装饰。

### 6.1 审查方法论（How — 怎么查）

以下方法贯穿所有校验项。每个方法有明确的操作步骤和停止条件。

**方法 A：调用链追踪**

1. 从 diff 中识别变更的函数/方法名
2. **向上追踪调用方**：grep 函数名 → 找到调用位置 → 再 grep 上层调用方。最多追 10 层，或到达路由入口/事件处理器层停止
3. **向下追踪被调用方**：读变更函数体内的函数调用 → grep 其定义。追踪到以下任一边界停止：框架 API（如 `setState`、`dispatch`、`emit`）、外部 HTTP/IO 调用、纯 UI 渲染函数
4. 记录完整链路：`[入口] → [中间层] → [变更函数] → [下游调用] → [终点]`

**方法 B：执行路径梳理**

1. 定位 diff hunk 所在的函数定义（不是整个文件）
2. 对该函数内每个条件分支（if/switch/三元/?./&&）：
   - 标注该分支对应 PRD 的哪条规则（`PRD R-XXX`）
   - 标注 `PRD 未覆盖`（PRD 没提但代码有的分支）
   - 标注 `新增分支`（本次 diff 新加的分支）
3. 重点检查：`PRD 未覆盖` 和 `新增分支` 的逻辑是否合理

**方法 C：多文件横向对比**

1. 从 TRD 规格的 `module` 字段获取同一需求的所有实现文件列表
2. 对每个文件，grep 目标逻辑的关键函数名/条件表达式
3. 逐文件比对：条件写法是否一致、处理逻辑是否一致、缺失的文件标记为缺陷
4. 注释掉的代码视为缺失

**方法 D：代码阅读范围**

| 场景 | 阅读范围 |
|------|----------|
| 默认 | diff hunk ± 30 行上下文 |
| 命中大文件热点（§5.4） | 读 diff hunk 所在的整个函数定义（从函数签名到结束括号） |
| 公共模块签名变更 | 读该函数的所有调用方的调用行（grep 获取） |
| 命中耦合热点（§5.4） | 读该文件的所有 export + 调用方的 import 行 |

**方法 E：隐式依赖推理**

校验项清单（6.2）覆盖的是已知模式。但最容易漏掉的 bug 来自**隐式依赖**——代码之间没有直接调用关系，却通过共享状态、全局资源、数据流、用户操作路径等间接耦合。这类问题无法穷举，需要 AI 从变更代码出发**主动推理**。

**核心思维：对每个变更，问"谁在我不知道的地方依赖了我改的东西？"**

推理步骤：

1. **识别变更的"输出面"**：这次修改改变了什么对外可观测的行为？不只是函数返回值，还包括：
   - 修改的数据会被谁读取？（共享状态、缓存、数据库、全局变量）
   - 修改的资源状态会影响谁？（音频播放状态、网络连接、锁、焦点）
   - 修改的内容会在哪里展示？（文案拼接、格式化、国际化 key）
   - 修改的时序会影响谁？（初始化顺序、异步完成时机、生命周期钩子顺序）

2. **追踪隐式消费方**：对每个"输出面"，用 grep/调用链追踪找到所有消费方，重点关注**不在本次变更范围内的消费方**——它们不知道你改了东西

3. **模拟用户操作路径**：从用户视角走一遍涉及变更的操作流程，在每个页面切换、返回、后台切前台的节点问：当前状态是否一致？

**典型案例**（用于建立推理直觉，非穷举清单）：

| 变更 | 隐式依赖 | 漏掉的 bug |
|------|----------|-----------|
| 修改了多语言某个 key 的文案 | 其他地方用这个 key 做字符串拼接或长度判断 | 拼接后的文案语义不通、截断异常、布局溢出 |
| 页面 A 新增了音频播放功能 | 路由跳转到页面 B 时没有暂停 | 两个页面的音频同时播放 |
| 详情页更新了数据并返回 | 列表页的数据源没有刷新 | 用户看到旧数据，以为没保存成功 |
| 修改了 Navigator pop 的参数 | 调用方依赖 pop 返回值做业务判断 | pop 返回 null 导致调用方空指针或逻辑分支错误 |
| 修改了公共日期格式化函数的输出格式 | 其他模块用格式化结果做正则匹配或解析 | 解析失败但不报错，静默产生错误数据 |
| 给 API 请求新增了一个必填参数 | 其他页面也调用同一个 API 但没传新参数 | 其他页面请求 400 报错 |
| 修改了全局主题色变量 | 某个组件硬编码了旧色值做对比判断 | 条件判断永远 false，UI 状态异常 |
| 修改了列表排序逻辑 | 其他地方用 index 定位特定数据项 | 取到错误的数据项 |

**执行要求**：
- 方法 E 在校验项清单之前执行——先自主推理，再用清单查漏补缺
- 发现的隐式依赖问题直接进入缺陷报告，与清单发现的问题同等对待
- 如果 architecture.md 存在，将推理结果与 §9.2（耦合热点）交叉验证，扩大排查面

### 6.2 校验项（What — 查什么）

#### 校验项裁剪（根据变更画像自动确定检查重点）

**禁止对所有校验项无差别平均用力。** 根据 Step 5 的变更画像（文件分层 + risk_notes + hotspot_matches），按下表确定每个校验项的执行深度：

| 变更画像信号 | 深度检查（逐项执行） | 快速扫描（有明显问题才深入） |
|---|---|---|
| UI 层文件变更 | A, D, K | B(仅边界条件), G(仅资源清理) |
| 状态管理层变更 | A, D, E | B(仅状态流转), F |
| 模型/数据层变更 | A, B, C, J | D, F |
| 公共模块/基类变更 | F, H | A, B |
| risk_notes 含"竞态"/"阻塞" | B(全部子项) | — |
| risk_notes 含"签名变更" | F, H | — |
| risk_notes 含"新条件分支" | A, B(条件完整性) | — |
| risk_notes 含"硬件/支付/认证" | B(环境降级+异步), I(安全) | E(幂等/重入) |
| risk_notes 含"数据模型字段变更" | C, J(兼容性) | F(消费方影响) |
| risk_notes 含"多文件实现不一致" | H(横向一致) | A(完整性) |
| hotspot_matches 非空 | 按 §5.4 增强项执行 | — |

**规则**：
- 同一校验项被多个信号命中 → 取最高深度（深度检查 > 快速扫描）
- 未被任何信号命中的校验项 → 快速扫描（30 秒内判断有无明显问题，无则跳过并在报告中标注"快速扫描未发现问题"）
- 条件触发项（I/J/K）仍按原有触发条件判断是否启用

#### 必查项（每次执行都检查）

**A. PRD 实现完整性 + 偏离检测**
- 每条验收标准是否有对应代码？
- **做的对不对？** PRD 要求 X，代码是否做了 X2（相似但不同的行为）
- **是否过度实现？** PRD 明确未要求 / 排除的场景，代码是否多做了（引入不必要复杂度和风险）

**B. 逻辑正确性 + 执行顺序 + 环境依赖**
- 条件判断完整？（`>= 1` vs `> 1`、遗漏 else/default）
- 边界条件？（空值、零值、首次加载、最大值、off-by-one）
- 状态流转正确？（是否有非法跳跃、遗漏状态）
- **阻塞前置风险**：依赖不可靠环境（网络、蓝牙、NFC）的阻塞操作是否放在关键路径前？
  - 典型：网络请求（可能超时 30s）写在识卡逻辑前 → 网络差时放卡无反应
- **异步竞态**：并发操作是否有正确的同步保证？先后依赖的异步调用是否正确 await？
- **环境降级**：网络断开、硬件异常、权限拒绝时是否有 fallback/超时？PRD 是否有降级要求？
- **资源顺序**：多资源获取/释放是否可能死锁？

**C. TRD 技术一致性**
- 使用了 TRD 指定的 API/方法？遵循架构约定？
- **TRD 偏离 ≠ 必定缺陷**：
  - 代码偏离 TRD **但满足 PRD** → 疑似文档漂移（置信度 4-6），标注"建议同步更新 TRD"
  - 代码偏离 TRD **且不满足 PRD** → 真实缺陷，正常置信度

**D. 状态管理 + 数据同步**
- 改了数据没刷新 UI？
- 乐观更新后请求失败 → UI 显示成功但数据未变？
- 缓存未失效 → 用户看到旧数据？
- A 页面改了共享数据，B 页面没刷新？
- 全局状态污染（A 页面操作意外影响 B 页面）？
- 状态泄漏（页面关闭后状态还在更新、幽灵回调）？
- **跨页面/路由副作用检测**（检测方法：找到变更文件中启动的长生命周期操作，追踪其停止/释放是否绑定在页面卸载路径上）：
  - 音频/视频播放：页面启动了播放 → 跳转其他页面时是否暂停/停止？返回时是否恢复？
  - 全局弹层/Toast/Dialog：页面打开了弹层 → 路由切换后弹层是否仍然显示？
  - 键盘/输入焦点：页面请求了焦点或打开了键盘 → 跳转后是否残留？
  - 后台任务：页面启动了轮询/长连接/下载 → 跳转后是否仍在运行消耗资源？
  - 滚动位置/手势锁定：页面锁定了滚动或注册了全局手势 → 跳转后是否释放？

**E. 幂等性 / 重入**
- 用户快速双击提交按钮 → 是否有防重复提交（debounce / loading 锁）？
- 页面重新进入 → 是否会重复初始化 / 重复注册监听？
- 网络重试 → 是否会导致重复操作（如重复扣款、重复创建）？
- 操作中途退出再进入 → 状态是否能正确恢复？

**F. 公共/耦合模块波及**
- 变更是否修改了公共模块（base class、共享组件、utils、公共服务）或耦合模块？
- 若是：import 反查所有消费方 → 逐个评估影响
- **三层检测**：
  1. **签名变更**：参数增删、类型变更、返回类型变更 → 编译/调用层面 break
  2. **默认值变更**：参数默认值改变 → 调用方未传参时行为静默改变
  3. **行为语义变更**（最易遗漏）：函数签名不变，但"做的事"变了。检测方法：
     - **业务语义变更**：读函数修改前后的逻辑，判断函数的"业务含义"是否改变（如"返回未完成的题目"→"返回所有题目"；"过滤无效数据"→"保留无效数据并标记"）
     - **业务假设检查**：对每个调用方，读其函数名和上下文，判断调用方是否依赖旧的业务语义（如调用方叫 `renderUnfinishedList`，依赖的是"只返回未完成"这个语义）
     - **返回值语义变更**：返回值的"含义"变了（如 null 从"未找到"变为"无权限"；空数组从"无数据"变为"已完成"）
     - **副作用变更**：函数新增或移除了副作用（如新增 emit 事件、新增状态写入、移除日志上报）
     - **条件逻辑变更**：新增 early return、修改过滤条件、变更排序规则等，导致同样输入产出不同结果
- 列出受影响模块清单和可能的副作用

**G. 资源泄漏 + 框架专项**
- 订阅/Timer/Stream/Controller 是否在 dispose/unmount 中清理？
- **资源生命周期配对检测**（检测方法：在变更文件中搜索"启动/获取"类 API 调用，逐个确认是否有对应的"停止/释放"调用绑定在页面卸载路径上）：
  - 音频/视频：play/start → pause/stop/release（dispose/unmount/deactivate 中）
  - 定位/传感器：watchPosition/startListening → clearWatch/stopListening
  - WebSocket/SSE：connect/open → close/disconnect
  - 通知/推送：requestPermission/subscribe → unsubscribe
  - 动画：forward/repeat → stop/dispose（AnimationController、requestAnimationFrame）
  - 文件/数据库：open → close
- Flutter: initState/dispose 配对、异步操作 mounted 检查、空安全、WidgetsBindingObserver 生命周期
- React: Hook 依赖数组完整性、useEffect 清理函数、卸载后 setState、ref 清理
- 后端: 事务边界完整性、连接/资源释放、并发安全
- 通用: 可空类型处理、文件句柄关闭
- **路由/导航场景专项**（变更涉及页面/路由组件时必查）：
  - 当前页面是否注册了全局监听（事件总线、广播、全局 Provider 写入）→ 跳转后是否移除？
  - 当前页面是否修改了全局配置（屏幕方向、状态栏样式、系统 UI 模式）→ 离开时是否恢复？
  - 当前页面是否持有独占资源（相机、麦克风、蓝牙连接）→ 离开时是否释放？

**H. 关键调用存在性 + 横向一致**
- "保存后刷新"类需求：逐文件检查回调中是否存在未注释的刷新调用
- 同一需求多处实现 → 每处都做了？逻辑一致？

#### 条件触发项（变更涉及时才检查）

**I. 安全/权限**（涉及多角色、认证、敏感数据时）
- 越权访问：用户 A 能否看到 / 操作用户 B 的数据？
- 敏感数据是否脱敏显示？
- 用户输入是否校验（防注入、防路径穿越）？
- Token 过期 / 鉴权失败的处理是否完整？

**J. 兼容性/迁移**（涉及 API 变更、数据结构变更、客户端版本时）
- 新字段上线，旧版本客户端解析是否安全（不崩溃）？
- 接口响应格式变更是否向下兼容？
- 旧数据迁移是否覆盖所有情况？
- 数据库 migration 是否可回滚？

**K. 交互反馈完整性**（涉及 UI 层变更时）
- 异步操作是否有 loading 态 / 骨架屏？
- 失败是否有错误提示 / 重试入口？
- 列表为空是否有空状态？
- 页面跳转路径是否正确？

### 6.3 缺陷判定

对每个发现的问题，按以下流程判定：

1. **[事实]** 代码实际做了什么？→ 引用文件、方法名、行号
2. **[规则]** PRD/TRD 要求做什么？→ 引用 requirement_id / specification_id
3. **[冲突]** 事实与规则的具体冲突点
4. **[触发场景]** 什么条件下触发此缺陷（越具体越好）
5. **[严重度]**
   - P0: 核心业务逻辑错误，影响主流程
   - P1: 重要功能缺失/错误，影响分支流程
   - P2: 边缘场景处理不当
6. **[置信度]** 基于证据链判定（1-10 分）：

   | 等级 | 分值 | 判定条件（三要素必须全满足） |
   |------|------|---------------------------|
   | **确凿** | 9-10 | ① 有代码事实（引用了文件:行号）+ ② PRD/TRD 明文规则与之冲突 + ③ 可描述具体触发场景 |
   | **高** | 7-8 | ① 有代码事实 + ② PRD/TRD 规则可合理推导出冲突（非明文但逻辑自洽）+ ③ 触发场景需特定条件 |
   | **中** | 4-6 | ① 有代码事实 + ② 但 PRD/TRD 未明确覆盖该场景（推断缺陷）或 ③ 依赖外部组件/服务的行为 |
   | **低** | 1-3 | 仅代码模式可疑 + 无 PRD/TRD 直接对照 + 可能是有意的设计选择 |

   **ASSERTION**：
   - `confidence: needs_confirmation` 的需求 → 该需求相关的缺陷最高只能给 6 分
   - 三要素中缺少任一 → 自动降一档（如有代码+有规则但无触发场景 → 从确凿降至高）
   - 仅凭"代码风格不好"或"可能有问题" → 不判定为缺陷，移至"观察项"

7. **[不确定性]** 影响判定的未知因素
8. **[修复建议]** 具体代码修复 + 验证方法
9. **[技术风险]** 修复可能带来的副作用

## Guardrails

⛔ 以下为全局红线，部分已在执行流程中内联重申。红线编号（GL1~GL9）与流程中的约束双向索引。

- ⛔ **GL1** 状态机 G1→G6 是 ASSERTION：G1 未完成 → 禁止开始检测。所有门禁必须逐一通过。→ 见 State Machine Gates
- ⛔ **GL2** 确凿证据原则：无证据不判定缺陷；不确定时用搜索工具获取真实代码再判断，禁止猜测。→ 见步骤 6
- ⛔ **GL3** 不确定性优先：依赖无法审查的外部组件 → 不武断判定，标注不确定性并降低 score。→ 见步骤 6
- ⛔ **GL4** 职责边界：后端接口正确性 → 后端职责不判定；非本变更引入的历史问题 → 不判定。→ 见步骤 6
- ⛔ **GL5** TRD 偏离 ≠ 必定缺陷：代码满足 PRD 但偏离 TRD 时 → 疑似文档漂移，置信度降低，不直接判定为缺陷。→ 见步骤 6
- ⛔ **GL6** 逐行审查：不要发现一个 bug 就停止，审查所有变更代码。→ 见步骤 6
- ⛔ **GL7** PRD/TRD JSON 不落盘：仅内存传递，不写入文件系统。→ 见步骤 3/4
- ⛔ **GL8** 不写业务代码：只输出报告，不修改任何源码。
- ⛔ **GL9** 报告按模板输出，不得简化或省略章节。→ 见步骤 6
