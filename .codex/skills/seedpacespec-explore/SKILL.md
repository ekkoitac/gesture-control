# seedpacespec-explore

## Goal

探索模式。用于**早期需求澄清**——当用户只有"一句话需求"、没有 Figma 稿或正式 PRD 时，从模糊想法 → 澄清需求 → 生成 PRD → 创建变更。

**定位**：`figma-to-prd` 的文字需求替代方案。两者最终都生成 PRD 到变更目录，后续流程一致。

**与 propose / guided 的衔接**：若用户在 **`$seedpacespec-propose`** 或 **`$seedpacespec-guided`** 中遇到「未找到 PRD」并选择「先去 explore 澄清」，通常会已有一个变更名 **`{name}`**（变更目录可能已由 CLI 创建）。建议在 explore 首轮对话或指令中带 **`{name}`**，使澄清后的 PRD 写入 `seedpacespec/changes/active-change/{name}/specs/`；PRD 就绪后再回到 **`$seedpacespec-propose {name}`** 或 **`$seedpacespec-guided {name}`**。意图分诊与「讨论性场景 → Plan / 普通对话」分流仅在 **本 skill 的阶段 0** 执行，propose/guided 不在入口处重复判断。

**探索风格**：

- **好奇，不说教** —— 自然生发问题，不照本宣科
- **多线程，非审讯** —— 铺开多个方向让用户跟上有共鸣的，不用问题链赶进单一路径
- **可视化** —— 有助于理清思路时，多用 ASCII 示意图（状态机、数据流、对比表等）
- **自适应** —— 跟进有意思的线索，新信息出现时可转向
- **耐心** —— 不急于下结论，让问题形状自己浮现
- **接地气** —— 相关时探索真实代码库，不要只空泛推演

**你可以做**：

- 探索问题空间 —— 澄清问题、挑战假设、重述/重构问题、寻找类比
- 调研代码库 —— 梳理现有架构、找到集成点、识别已用模式、暴露隐藏复杂度
- 比较需求方向 —— 头脑风暴多种功能范围、做对比表、勾勒权衡
- 暴露风险与未知 —— 指出可能出问题的点、发现理解缺口、建议 spike 或专项调研

## State Machine Gates

⛔ 约束: 任何门禁未完成 → 只能向用户确认/请求输入，不得创建文件或写入内容。

| #  | 状态                | 完成条件                                                                  | 未完成时行为           |
|----|---------------------|---------------------------------------------------------------------------|------------------------|
| G0 | `intent_triaged`    | 已确认用户输入是"真实需求"（或被引导切到 Plan/普通对话后流程结束）        | 阶段 0 意图分诊 + 向用户确认/请求输入 |
| G1 | `input_understood`  | 用户意图已澄清（模糊想法 → 具体需求）                                     | 向用户确认/请求输入 澄清需求   |
| G2 | `scope_converged`   | 需求范围已发散/收敛，用户认可方向                                          | 继续探索或 向用户确认/请求输入 |
| G3 | `creation_decided`  | 用户决定是否创建变更（是/否/再想想）                                       | 向用户确认       |
| G4 | `change_created`    | 变更目录已创建，PRD 已写入 specs/                                          | 执行创建               |

⛔ 约束: 禁止编造跳过理由——不得以任何自创名义（精简流程、复用已有产物、简化模式等）跳过门禁。如果你发现自己正在向用户提议跳过某个门禁——立即停止并按正常流程执行。

## 执行流程

### 阶段 0：意图分诊（→ G0）

⛔ 约束: 进入 G1 之前，必须先识别用户输入是"真实需求"（想最终落地一个功能）还是"讨论性场景"（只想比较/评估/学习/理解）。后者不应进入 spec 流程，应引导切到 Plan 模式或作为普通对话回答——避免用户被迫看一堆 PRD/TRD/tasks。

#### 真实需求信号（命中其一即可直接进入 G1，不必 向用户确认/请求输入）

- 含"做/加/实现/构建/上线"+ 名词性产物（"做个登录"、"加个分享按钮"、"实现实时协作"）
- 含"我想要/我们要"+ 行为目标
- 描述包含明确的目标用户、使用场景或验收意图
- 已经在描述功能边界、范围取舍

#### 非需求信号（命中且无真实需求信号 → 必须向用户确认/请求输入 分诊）

| 类别                  | 用户语言信号                                                           | 真实诉求               | 推荐路径    |
|-----------------------|------------------------------------------------------------------------|------------------------|-------------|
| **方案对比/选型**     | "X 和 Y 哪个好"、"该用 A 还是 B"、"对比一下"、"X vs Y"                 | 想要判断，不打算落地   | Plan 模式   |
| **架构论证/可行性**   | "这样设计行不行"、"会不会有问题"、"评估下方案"、"是不是更好"           | 想要 review，不一定做  | Plan 模式   |
| **概念学习/技术问答** | "X 怎么用"、"X 的原理"、"X 是什么"、"如何理解 X"                       | 求知，不做产品决策     | 直接对话    |
| **代码现状理解**      | "这块代码怎么走"、"为什么这样写"、"X 是怎么实现的"、"帮我看下这段"     | 探索代码，不做需求     | 直接对话    |
| **风险/影响评估**     | "改这个会影响哪些"、"这么做有什么坑"、"会不会有副作用"                 | 评估，不一定改         | Plan 模式   |

#### 处理流程

1. **真实需求信号清晰** → G0 通过，直接进入 G1
2. **非需求信号清晰** → 必须向用户确认/请求输入：

   **用户可见文案**：
   - question: "你的描述更像「{命中类别}」，不一定要走 spec 流程（PRD → TRD → tasks）。怎么继续？"
   - option 1: label="切到 Plan 模式讨论" description="开放讨论，不生成任何文档，不创建变更"
   - option 2: label="就在这里聊" description="作为普通对话直接回答，不切换模式"
   - option 3: label="不，这是真实需求" description="确实想做出来，继续走 explore 流程"

3. **信号混合或模糊**（既像需求又像讨论）→ 单问一句确认目标：

   **用户可见文案**：
   - question: "想确认一下你的目标："
   - option 1: label="想最终落地一个功能" description="走 explore → 生成 PRD → 创建变更"
   - option 2: label="先讨论清楚要不要做或怎么做" description="切到 Plan 模式开放讨论"

#### 用户选项处理

- **「切到 Plan 模式讨论」/「先讨论清楚要不要做或怎么做」** → 调用 **SwitchMode** 工具，`target_mode_id="plan"`，`explanation` 简述识别到的非需求场景。SwitchMode 切换由用户确认；切换后本 skill 流程**到此结束**，不进入 G1，不创建任何文件。
- **「就在这里聊」** → 作为普通对话回答用户问题，本 skill 流程**到此结束**，不进入 G1。
- **「这是真实需求」/「想最终落地一个功能」** → G0 通过，进入 G1。

⛔ 约束: G0 在每次 explore 启动时**只执行一次**——一旦通过，后续阶段不得反复盘问意图。
⛔ 约束: 用户在 G0 选「切 Plan」或「就在这里聊」时，严禁创建任何文件、严禁继续 G1+ 流程。
⛔ 约束: 不得以"先帮用户搭一下框架"、"顺手生成 PRD 让用户参考"等理由跳过 G0 引导切走的决定。

### 阶段 1：需求澄清（→ G1）

用户输入模糊想法，通过自然对话澄清：

- 核心用户是谁？使用场景是什么？
- 功能边界在哪？哪些是必须的、哪些是可选的？
- 有没有现有系统的约束或集成要求？
- 验收标准是什么？

不要一次性问完，根据用户回答自然延伸。

### 阶段 2：需求发散与收敛（→ G2）

⛔ 约束: 本阶段只发散**需求层面**的方向（功能范围、用户场景、优先级取舍），不做技术方案选型。技术方案（如 JWT vs Session、WebSocket vs SSE）留给 propose 的 design 阶段。

需求基本清晰后，对功能范围提供多种方向对比：

```
| 方向     | 包含功能                     | 复杂度 | 适用场景       |
|----------|------------------------------|--------|----------------|
| 最小可用 | 邮箱登录 + 忘记密码         | 低     | MVP / 内部工具 |
| 标准     | + 第三方登录 + 注册          | 中     | C 端产品       |
| 完整     | + 多设备策略 + 安全审计      | 高     | 企业级产品     |
```

让用户选择或混合，直到需求范围明确。

### 阶段 3：提议创建变更（→ G3）

当需求清晰时，**向用户确认/请求输入**：

> 需求已澄清，是否创建变更并生成 PRD？
>
> A）创建变更并生成 PRD → 进入下一步
> B）继续探索 → 回到阶段 2
> C）结束探索 → 不创建

### 阶段 4：创建变更与生成 PRD（→ G4）

⛔ 约束: G3 前禁止落盘——需求未清晰前，不创建任何文件。用户确认后才创建。

用户选 A 后：

**4.1 确定变更名**
- 根据需求推导 kebab-case 名称
- 向用户确认：`"建议变更名: 'feat-login'，是否使用？"`

**4.2 创建变更目录**
```bash
seedpacespec new change "<name>"
```

**4.3 生成 PRD 并写入 specs/**

⛔ 约束: PRD 只回答"做什么、为谁做、验收标准"，不包含技术方案选型。

按以下结构生成，文件路径：`seedpacespec/changes/active-change/<name>/specs/<name>-prd.md`

```markdown
# [功能名称] PRD

## 背景与目标

[为什么要做这个功能，解决什么问题，目标用户是谁]

## 需求清单

| ID  | 用户故事 | 验收标准 | 优先级 | 需求强度 |
|-----|----------|----------|--------|----------|
| R01 | ...      | ...      | P0     | SHALL    |
| R02 | ...      | ...      | P1     | SHOULD   |

## 需求详述

### R01 [需求标题]

**用户故事**: 作为[角色]，我希望[行为]，以便[价值]

**场景**:

- WHEN [前置条件] THEN [预期结果]
- WHEN [异常条件] THEN [异常处理]

### R02 [需求标题]

...

## 非功能性需求（可选）

[性能、安全、兼容性等约束]

## 开放问题（可选）

[探索阶段遗留的待确认事项]
```

- 使用 SHALL/SHOULD/MAY 表达需求强度
- 每个需求至少一个 Scenario（WHEN/THEN）
- **需求编号**：格式 `R01`、`R02`、…（两位数字，从 01 起连续递增）；下游 TRD 和 tasks 通过此编号回溯对应需求

**4.4 确认完成**

输出：
```
✅ 变更 "{name}" 已创建，PRD 已生成！

📁 变更目录: seedpacespec/changes/active-change/{name}/
📄 PRD 文件: specs/{name}-prd.md

下一步：
- 生成设计文档？执行 $seedpacespec-propose "{name}"
- 想逐步引导决策？执行 $seedpacespec-guided "{name}"
- 直接实现？执行 $seedpacespec-apply-change（需先 propose 或 guided 生成 tasks）
```

## Guardrails

⛔ 以下为全局红线，部分已在执行流程中内联重申。

- ⛔ 意图分诊优先（G0）：进入 G1 之前，命中"非需求信号"必须向用户确认/请求输入 分诊；用户选择切 Plan 或普通对话时，调用 SwitchMode 后立即终止 skill，不得继续 G1+。→ 见阶段 0
- ⛔ 不实现代码：explore 只生成文档（PRD），绝不编写业务代码。若用户要求实现，提醒先完成 explore → 创建变更 → 再 `$seedpacespec-apply-change`。
- ⛔ PRD 不含技术方案：PRD 是需求澄清产物，不包含技术选型、架构设计、实现细节。这些留给 propose 的 design/TRD 阶段。
- ⛔ G3 前禁止落盘：需求未清晰前，不创建任何文件。
- ⛔ 用户确认后才创建：任何文件写入前必须向用户确认/请求输入 获得明确许可。
- ⛔ 禁止跳过门禁：不得以任何自创名义跳过 G0-G4 中的任何一个。
