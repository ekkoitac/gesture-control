# 反馈记忆协议（Memory Protocol）

## 开关

读取 `seedpacespec/config.yaml`：
```yaml
memory:
  enabled: true
```
- `memory.enabled` 不为 `true`（包括字段不存在、config.yaml 不存在）→ 跳过所有记忆相关操作
- init 模板默认开启（`enabled: true`）

## 目录与文件

```
seedpacespec/memory/
  team-feedback.md      # 团队级（提交到 git）
  personal-feedback.md  # 个人级（.gitignore）
```

文件不存在时：读取跳过；首次写入时自动创建。

## 条目格式

```markdown
### [✅|❌|⚖️] {一句话结论}
- **原因**: 为什么这是对的/错的/为什么这样选
- **适用场景**: 什么情况下这条记忆应该影响决策
- **日期**: YYYY-MM-DD
```

三种类型：
- ❌ 纠正：AI 方案被用户修改，记录错在哪、正确做法
- ✅ 肯定：AI 方案被认可且包含值得记住的判断，记录什么场景下这样做是对的
- ⚖️ 决策：用户在两个合理选项中做了选择，记录为什么选 A 不选 B

## 不记录什么

- 代码能发现的（架构、文件路径、模块关系）
- git 能查的（版本历史、谁改了什么）
- 文档已记的（需求内容、技术方案细节 → PRD/TRD/design.md）
- 机械操作（列文件、跑命令、搜索）
- 显然的确认（用户只说"可以"/"好"之类的过一个常规步骤）

**只记录**：用代码和文档推断不出来的判断偏好——遇到岔路口往哪走。

### 判断示例

| 场景 | 记录？ | 类型 | 理由 |
|------|--------|------|------|
| 用户说"组件级状态用 useState，跨模块才上 Redux" | ✅ | ⚖️ | 代码能看到当前用了什么，但看不出"什么场景该用什么"的决策原则 |
| 用户说"这个接口返回值加个 total 字段" | ❌ | — | 具体需求细节，TRD/PRD 里已有 |
| 用户说"别用 enum，这个项目统一用 union type" | ✅ | ❌ | 编码偏好，代码能看到现状但看不出"统一要求" |
| 用户确认了 AI 提出的"事件驱动而非轮询"方案 | ✅ | ✅ | 架构取舍偏好，值得在后续设计中复用 |
| TRD 改了4轮，每轮都在砍复杂度 | ✅ | ❌ | 提炼为"用户偏好最小化方案"，不逐条记每轮改了啥 |

## 读取时机

在 TRD/design **生成前**（Step 3 加载上下文阶段）：
1. 读取 `seedpacespec/memory/team-feedback.md`（若存在）
2. 读取 `seedpacespec/memory/personal-feedback.md`（若存在）
3. 将条目作为**生成约束**，生成方案时参考这些偏好
4. **记忆是线索而非事实**——引用记忆中涉及具体路径、函数名、技术选型的条目时，必须先验证其当前是否仍然成立（读文件/grep 确认），再作为约束使用。若验证发现已失效，删除或更新该条目。

## 收集时机

在**确认循环通过后**（用户确认 TRD/design 不需修改后），执行以下判断：

### 纠正收集（❌）

**核心思路：从多轮修改中学习用户的决策思路，而非每次改一条记一条。**

用户确认通过时，回顾整个确认循环（v1→vN）：
1. **统计修改轮次**：用户选了几次"需要修改"？
2. **提取修改方向**：每轮修改在纠正什么？是否指向同一个倾向？
3. **学习决策思路**：从所有修改中归纳出用户的决策倾向，而不是逐条记录每次修改的具体内容

**写入规则**：
- 只在确认通过后写入（不是每轮修改都写）
- 条目应是"用户倾向于 X 而非 Y"的形式，附带推导依据
- 能合并为一条模式的，合并为一条；**合并不了的（指向不同维度的偏好），每个维度单独一条**
- `适用场景` 字段要写清楚：下次生成什么类型的方案时应该直接按这个倾向来

**示例**：用户修改了4轮 TRD，v1 用了全局状态管理被改，v2 拆了太多文件被改，v3 加了抽象层被改，v4 通过。
- ❌ 记录：「用户偏好最小化方案——避免全局状态、避免过度拆分文件、避免提前抽象，能用局部方案解决的不要上全局方案」
- 而**不是**记4条：「不要用全局状态」「不要拆太多文件」「不要加抽象层」「v4 通过了」

### 肯定收集（✅）

用户确认通过时：
- 判断用户的确认是否带有明确的肯定信号（如"对就该这样"、"这个方案很好"、"比之前好"、语气明显正面）
- 或方案中包含了某种取舍判断（合并 vs 拆分、激进 vs 保守等）且用户通过了
- 是 → 写入 ✅ 条目

### 决策收集（⚖️）

用户在讨论中主动解释了"为什么选 X 不选 Y" → 写入 ⚖️ 条目

### 写入规则

- **必须用文件写入工具（文件写入/编辑操作）将条目追加到记忆文件中**——仅在对话中展示但不写入文件视为未完成
- 默认写入 `team-feedback.md`
- 仅当用户明确说"这是我个人偏好"或类似表述时写入 `personal-feedback.md`
- 写入前检查是否已有类似条目 → 有则更新而非重复添加
- 每次确认循环最多写入 1 条（避免过度记录）

## 推翻机制

当新条目与已有条目矛盾时：
- 在旧条目的标题前加 `~~` 删除线标记作废
- 新条目末尾注明：`> 取代: {旧条目标题}`

## 写入前确认

写入记忆条目前，**简短告知用户**即将记录的内容（一句话），无需等待确认即可写入。示例：
> 📝 记录反馈：这个项目中 service 层偏向合并而非拆分。

用户如果不同意可以随后让 AI 删除。
