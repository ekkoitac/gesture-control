# seedpacespec-design-module

## 触发命令

```
$seedpacespec-design-module <模块描述>
```

**示例**：
- `$seedpacespec-design-module 动画控制器，多个 lottie 动画按顺序衔接，支持并发控制和中断恢复`
- `$seedpacespec-design-module 文件上传引擎，支持分片、断点续传、加密`
- `$seedpacespec-design-module 离线缓存同步系统`

参数为模块的自然语言描述，也可不带参数直接 `$seedpacespec-design-module`，AI 会追问需求。

---

## 定位

独立于 TRD 生成流程，专门为**无 PRD、无视觉稿的复杂逻辑模块**输出设计方案。

**典型场景**：
- "我要做一个动画控制器，多个 lottie 动画衔接，需要并发控制"
- "帮我设计一个离线缓存同步引擎"
- "音频播放器需要支持插件扩展，帮我出个方案"
- "设计一个任务调度器，支持优先级队列和重试"

**不适用**：
- 有 PRD 的完整功能 → 用 `$seedpacespec-trd-generator`
- 简单工具函数或单类封装 → 直接写代码，不需要设计方案

---

## 输入

用户用自然语言描述模块需求。没有固定格式，AI 从描述中提取以下信息：

| 要素 | 说明 | 示例 |
|------|------|------|
| **模块目标** | 这个模块要解决什么问题 | "控制多个 lottie 动画按顺序/并行播放" |
| **核心能力** | 模块要支持哪些关键操作 | "播放、暂停、跳转、插入、取消" |
| **变化点** | 哪些部分未来可能变化或需要扩展 | "动画类型可能新增 spine、视频片段" |
| **约束** | 性能、平台、兼容性等限制 | "Flutter 端，动画切换间隔 < 16ms" |
| **已有上下文** | 项目中已有的相关代码、架构约定 | "项目用 Bloc 管状态，有 architecture.md" |

## Workflow

### Step 1: 场景甄别 + 理解需求 + 加载上下文

**1a. 场景甄别（防误用，向用户确认/请求输入）**

> **两种调用方式**：
> - **用户直接调用** `$seedpacespec-design-module` → 走下方 向用户确认/请求输入 甄别
> - **被其他 skill 内部引用**（trd-generator Step 7 / propose design 步骤遇到复杂模块时,自动按本 skill Step 3 的思考粒度展开）→ 不走本步甄别,直接使用 Step 3 的输出结构

用户直接调用时,先判断是否适合本 skill。用 向用户确认：

```
你描述的模块属于哪种情况？

A) 独立复杂模块 — 没有 PRD,是基于口头描述的独立系统/公共组件/引擎
   (如：动画控制器、离线缓存引擎、插件体系、状态机框架)
   → ✅ 适合本命令,继续

B) 某个需求的技术方案 — 有 PRD,是某个功能需求的实现方案
   → ❌ 应该用 $seedpacespec-trd-generator 生成 TRD
   (如果你在 propose/trd-generator 流程中遇到了复杂模块,
    不需要手动调本命令——那些流程会自动按本 skill 的思考粒度展开)
```

- 选 **A** → 继续 Step 1b
- 选 **B** → 提示用户「建议使用 `$seedpacespec-trd-generator`」,结束本 skill

**1b. 理解需求**

1. 从用户描述中提取上述要素，列出关键信息确认理解无误
2. 如果用户提供了项目路径或提到了 `architecture.md` → 读取，后续设计与项目架构保持一致
3. 如果用户提到了已有代码（"参考 XX 模块"）→ 读取，了解已有模式

### Step 2: 复杂度判定

对用户描述做以下扫描，逐条评估并输出判定表：

| # | 检查项 | 怎么判断 | 本次结果 |
|---|--------|---------|---------|
| 1 | **协作实体数** | 完成该模块需要的类/组件 ≥ 3 个，且有调用或数据传递关系 | ✅/❌ |
| 2 | **抽象层级** | 需要接口/抽象类 + 多个实现来屏蔽变化 | ✅/❌ |
| 3 | **异步编排深度** | 异步步骤 ≥ 3 步，且有顺序依赖、分支或失败回退 | ✅/❌ |
| 4 | **状态复杂度** | 状态 ≥ 3 个，转移条件非 trivial | ✅/❌ |
| 5 | **扩展点需求** | 描述中有"后续新增"、"可配"、"插件"等可扩展性要求 | ✅/❌ |
| 6 | **跨模块副作用** | 操作会触发其他模块的状态变更、事件、UI 刷新 | ✅/❌ |

**命中 ≥ 2 项** → 输出完整设计方案（Step 3 全量执行）

**命中 0-1 项** → 用 向用户确认/请求输入 告知用户："这个模块复杂度不高，建议直接写职责说明 + 接口定义，不需要完整架构设计。要继续完整方案还是简化版？"
- 用户选完整 → Step 3 全量
- 用户选简化 → 只输出 Step 3 中的第 3、5 节（职责表 + 接口签名）

### Step 3: 生成设计方案

读取设计写法标准：

```
读取文件: .codex/skills/seedpacespec-trd-generator/references/frontend/ui-component-guide.md
→ 定位到 "### 复杂逻辑系统的写法" 章节，按其中的写法标准执行
```

按以下结构输出设计方案文档。每一节都必须用**同一个模块**的例子贯穿，不得在不同节之间切换例子。

### Step 4: 保存

方案生成后，询问用户保存位置。提供默认选项：

1. **有变更目录** → 保存到 `seedpacespec/changes/active-change/<变更名>/designs/{模块名}-design.md`
2. **无变更目录** → 保存到用户指定路径，或当前工作目录下 `{模块名}-design.md`
3. **不保存** → 仅在对话中展示

## 写作标准

本 skill 的设计方案写作标准与 `seedpacespec-trd-generator` 中"复杂逻辑系统的写法"完全一致。

**核心要求**（摘自 `references/frontend/ui-component-guide.md`）：

1. **同一例子贯穿所有章节** — 类图、模块图、职责表、伪代码、协作流程用同一个模块
2. **设计模式必须有伪代码** — 不能只说"用策略模式"，要写到"注入点在哪、切换时怎么处理、异常怎么隔离"
3. **协作流程要有深度** — 不是"A 调 B 调 C"，而是写清每步的输入输出、失败回退、状态流转
4. **边界与异常** — 并发、资源释放、状态恢复、降级策略
5. **白话解读** — 伪代码块后必须跟中文白话，解释编排逻辑和兜底策略

**禁止**：
- 只有骨架没有深度（"AnimationManager — 管理动画"）
- 过度设计简单模块（一个 boolean 能解决的别上状态机）
- 类图没有模式标注
- 伪代码没有白话解读

| 场景 | 用哪个 |
|------|--------|
