## 技术选型与决策

> **用途**：统一管理所有设计抉择，TRD 直接引用 D 编号采纳决策而不重新评估，避免设计-实现分裂。Reviewer 在此判断选型合理性。
>
> **本节统一管理所有设计抉择**,用 `[架构级]` / `[模块级]` 标签区分粒度。
> 每条决策编号 D01~Dnn,后续 TRD 可直接引用(如"采纳 D03")。
>
> **[架构级]**：影响整个系统或整个变更的方向性选型(微内核 vs 单体 / SSR vs CSR / DDD vs 贫血 / 事件驱动 vs 请求-响应 等)。

>
> 若本次变更不涉及架构级决策,不需要硬凑——直接从 D01 开始写模块级决策即可。

## 模块总览

> **用途**：回答"本次变更拆了哪些模块、各自管什么、怎么协作、归属哪一层"。TRD 按此划分展开"方案"子章节，实现者据此理解系统结构。
>
> **分层来源**（按优先级）：
> - architecture.md 已有明确分层 → 标明每个模块归属哪个全局层
> - architecture.md 不存在，但项目已有代码 → 基于代码目录结构和职责分布推导出通用层，在本节开头简述划分依据
> - 全新项目（无 architecture.md 也无已有代码）→ 基于目标技术栈的标准通用层组织模块。在本节开头声明：「本项目为全新搭建，基于 {技术栈} 标准分层架构，采用以下通用层：{列表}」。常见参考：
>   - Web 前端：表现层 / 应用层 / 服务层 / 基础设施层
>   - 后端服务：API 层 / 服务层 / 领域层 / 数据层 / 基础设施层
>   - 移动端：表现层 / 业务逻辑层 / 数据层 / 平台适配层
>   - 其他技术栈：参照软件设计理论选取合适分层并说明理由
>
> **子层自由度**：design 可按业务语义自由划分子层（如"公共容器层"、"模板层"、"宿主层"等），也可新增架构抽象层（如"防腐层"、"适配层"、"网关层"等）。每个层/子层标注其在软件设计理论中归属的通用层即可。新增层须在"技术选型与决策"中说明引入理由。
>
> 用 ASCII 图画出模块关系,图中标注每个模块所属的层/子层。

### 模块职责与边界

| 模块 | 所属层/子层 | 归属通用层 | 职责 | 输入 | 输出 | 与周边关系 | 扩展点 |
|------|-----------|-----------|------|------|------|-----------|-------|
| 模块 A | Page | ... | ... | ... | 依赖 Hook X | ... |
| Hook X | Hook | ... | ... | ... | 消费数据模块 Z | ... |
| 数据模块 Z | Repository | ... | ... | ... | ... | ... |

> "归属通用层"列的值来源：有 architecture.md 时必须是其中定义的层名；无 architecture.md 时使用本节开头声明的通用层名（必须是软件设计理论中的标准通用层，不可用项目特有命名）。
> 扩展点：接口预留、插件位、配置位、事件钩子等，让 reviewer 看懂"后续怎么加东西"。

## 核心模块设计

> **用途**：design.md 的核心价值——记录每个模块的设计推理（为什么存在、边界、协作、取舍），TRD 直接引用这些推理不重复展开。缺少此节则 TRD 需自行推断 why，增加偏离风险。
>
> 不写代码实现(那是 TRD 的事),只回答:
> 1. 这个模块为什么要存在?(不拆会怎样?)
> 2. 它的边界在哪里?(什么不归它管?)
> 3. 它和谁协作?数据怎么流?(ASCII 图或一段话)
> 4. 有什么设计上的取舍?(选了什么,放弃了什么,为什么)
