# seedpacespec-update-task


## Plugin Protocol

执行本 skill 前，先读取 `seedpacespec/config.yaml` 的 `plugins` 配置。

**override**：若 `plugins.override.update-task` 存在 → **停止执行本文件**，改为加载指定文件。

**slots**：本 skill 暴露以下插槽，可在 `config.yaml` 的 `plugins.slots` 中挂载自定义 skill：

| 插槽名 | 位置 | 类型 | 说明 |
|--------|------|------|------|
| `after_trd_confirmed` | TRD 更新确认后（步骤 5.2 之后） | hook | 可用于 TRD 后处理、额外文档同步 |
| `after_tasks_confirmed` | tasks 更新确认后（步骤 6 之后） | hook | 可用于任务后处理、通知 |

**执行规则**：流程中遇到 `<!-- slot:名字 -->` 标记时，检查 `plugins.slots` 是否有匹配的 `skill: update-task` + `slot: 名字` 条目 → 有则加载 `run` 指定的文件执行，完毕后回到主流程继续 → 无则跳过。

若无 `plugins` 配置 → 正常执行，无任何变化。

## Goal

在已有变更中**添加新功能**或**修改已有设计方案**，自动联动更新以下产物：
- **TRD**：新增技术方案 或 修改已有方案（组件分层、状态归属、逻辑策略、接口契约等）
- **design.md**：新增/修改模块结构、技术决策
- **tasks.md**：新增任务项 或 修改/重置已有任务项

**适用场景**：

| 场景 | 模式 | 示例 |
|------|------|------|
| 遗漏需求、需求变更、追加功能 | **新增模式** | "加一个导出 PDF 功能" |
| 实现中发现设计问题、想换方案 | **修改模式** | "把状态管理从 Redux 改成 Zustand"、"这个组件的分层不合理要调整" |
| 拆分大任务 | **新增模式** | "把任务 3 拆成子任务" |
| apply 中暂停，需要改设计再继续 | **修改模式** | "TRD 里方案章节的 xx 方案要改" |

**核心原则**：
- **融入式增量更新**：新内容融入现有章节，不机械插入新章节；修改时就地重写受影响段落
- **联动更新原则**：修改一处后检查关联章节，保持产物间一致性
- **用户反馈驱动**：用户给出修改意见后，合理更新 TRD 相关部分
- **修改模式的任务重置**：若修改了 TRD 中已实现任务对应的方案，受影响的 `[x]` 须改回 `[ ]`

⛔ 约束：执行本命令对应流程时，**无论如何不得**生成或修改业务仓库中的应用源码（`src/` 等实现文件）；仅允许维护 `seedpacespec/changes/active-change/<name>/` 内产物文档；业务代码须通过 **`$seedpacespec-apply-change`** 完成。

## State Machine Gates

执行前必须依次通过以下状态门禁。**任何状态未完成 → 只能向用户确认/请求输入，不得进入后续步骤。**

| # | 状态 | 完成条件 | 未完成时行为 |
|---|------|---------|-------------|
| G1 | `change_selected` | 目标变更已确定 | 请求用户选择 |
| G2 | `features_parsed` | 用户提供了功能点描述，AI 已解析出功能条目 | 向用户询问功能点 |
| G3 | `context_loaded` | 所有产物文档已阅读（architecture.md、design、TRD、tasks、PRD、proposal）；R 编号已归一化 | 逐一读取 |
| G4 | `user_confirmed_scope` | 用户确认解析结果（选 A 继续 / B 调整 / C 取消→流程终止） | 向用户展示并确认解析结果 |
| G5 | `trd_confirmed` | TRD 已更新 **且** 用户确认正确（确认循环通过）；C 取消→流程终止 | 向用户确认循环 |
| G6 | `design_updated` | design.md 已按需更新（仅 G5 通过后执行） | 更新 design |
| G7 | `tasks_confirmed` | tasks.md 已更新 **且** 用户确认（确认循环通过） | 向用户确认循环 |

**状态推进单向** G1→G2→…→G7，不得跳跃。G4 选 C 或 G5 选 C 时流程提前终止。

### 恢复协议（未同步产物检测）

**用途**：当 G1 完成后检测到产物未同步状态时，本协议决定从哪个门禁恢复执行。下方"G2~G7"对应「State Machine Gates」表和「执行流程」章节中的步骤编号（如"G5"= 执行流程 → 步骤 5 自动更新产物）。恢复协议**不是独立流程**——它只决定从哪里开始，进入后走的是正常执行流程。

G1（选择变更）完成后，检查 TRD 的 `## 变更记录` 是否存在新增条目但 design/tasks 尚未同步更新。

**若检测到未同步 → 强制 向用户询问意图**，不得自行推断：

> **📋 同步恢复 用户可见文案模板**：
> - question: "检测到上次对技术方案的修改尚未同步到任务列表，如何处理？"
> - option 1: label="继续同步上次的修改" description="将上次未完成的方案修改同步到设计文档和任务列表"
> - option 2: label="忽略上次，开始新的修改" description="跳过未同步的内容，直接开始新的修改"

**选项 A（继续同步）**：
1. G2 跳过——功能点从 TRD 变更记录中提取，不再重新询问
2. G3 启用**精简阅读**：
   - **必读**：TRD 全文 + tasks.md + design.md（树状图章节）
   - **可跳过**：proposal.md、PRD、architecture.md（除非涉及新架构层）
3. 从断点 gate 继续：
   - TRD 已更新但用户未确认 → G5（确认 TRD）
   - TRD 已确认但 design 未更新 → G6（更新 design）
   - design 已更新但 tasks 未更新 → G7（更新 tasks）

**选项 B（全新请求）**：正常走 G2→G7，视为首次执行。

**未检测到未同步状态** → 视为首次，走完整 G1→G7。

⛔ 约束 [GL2]: 禁止编造跳过理由——AI 不得以任何自创名义（如恢复协议、追加模式、精简流程、复用已有产物、简化模式等）直接跳过门禁。所有门禁必须逐一通过并产出可验证输出。如果你发现自己正在向用户提议跳过某个门禁——立即停止并按正常流程执行。

## 执行流程

**输入**：功能点描述（支持自然语言描述或结构化格式，支持多条用换行或分号分隔）。

### 步骤 1: 选择目标变更（→ G1）

- 若已提供变更名则使用；否则运行 `seedpacespec list --json` 获取进行中的变更列表。
- 使用 **Codex 用户确认/输入流程** 让用户选择要添加任务的变更。
- 确认当前使用的变更：`<变更名>`。

### 步骤 1.5: 加载反馈记忆

按 `.codex/skills/_shared/memory-protocol.md` 的"读取时机"执行：读取 `seedpacespec/config.yaml` 的 `memory.enabled`，为 `true` 时读取 `seedpacespec/memory/team-feedback.md` 和 `personal-feedback.md`（若存在），作为后续 TRD/design 更新的生成约束。

### 步骤 2: 获取用户输入并识别模式（→ G2）

- 若未提供描述，使用 **Codex 用户确认/输入流程**（开放式）提问：
  > "请描述要添加的功能或要修改的设计方案："
- 解析输入，**自动识别模式**：
  - **新增模式**：描述中含"加/新增/追加/补充"等意图，或描述的是 TRD 中不存在的功能 → 走新增流程
  - **修改模式**：描述中含"改/调整/换/重构/优化"等意图，或指向 TRD 中已有的章节/方案 → 走修改流程
  - **混合**：同时有新增和修改 → 拆分为多个条目，分别标记模式
  - **无法确定**：关键词冲突或意图模糊 → **必须向用户确认/请求输入** 确认

> **📋 操作模式 用户可见文案模板**：
> - question: "请确认你想要进行的操作类型："
> - option 1: label="新增功能点" description="在已有方案中增加新的功能需求和对应任务"
> - option 2: label="修改已有方案" description="调整现有技术方案的设计或实现方式"
> - option 3: label="两者都有" description="既有新增也有修改，我来分别说明"

- 提取功能条目，每条标注模式（`[新增]` 或 `[修改]`）。识别完成后在步骤 4 中展示识别理由，供用户核实。

### 步骤 3: 解析功能点并评估影响范围（→ G3）

⛔ 约束 [GL3]: G3 未完成 → 不得进入 G4（确认）或 G5（更新产物）。

- 对每个条目进行解析，识别：
  - **模式**：新增 / 修改
  - 所属业务模块
  - 影响的技术层（UI/数据/状态/模型等）
  - 是否涉及新增文件/修改现有文件
  - 与现有功能的关系（独立/依赖/扩展）
  - **[修改模式额外]**：定位 TRD 中受影响的章节、受影响的已有任务

- **完整阅读**现有产物获取上下文（**每一项都必须读取，不得遗漏**）：
  - **项目根目录 `architecture.md`** — 架构分层、编码约定、模块依赖
  - **`seedpacespec/global-specs/` 目录**（若存在）— 项目背景、整体需求、业务规则
  - `seedpacespec/changes/active-change/<变更名>/design.md` — 现有模块结构、技术决策、已确认文件树
  - `seedpacespec/changes/active-change/<变更名>/tasks.md` — 现有任务状态
  - **变更目录 `trds/` 下的 TRD 文件** — 现有技术方案全貌
  - `seedpacespec/changes/active-change/<变更名>/specs/*-prd.md` — 需求上下文
  - `seedpacespec/changes/active-change/<变更名>/proposal.md` — 变更目标与范围

**3.1 R 编号识别与归一化（硬性约束）**

- 仅将「`R` + 数字」视为需求编号，兼容：`R1`、`R01`、`R-01`、`R 01`、`R_01`（大小写不敏感）
- 生成双标识：
  - `rawId`：PRD/TRD/tasks 中的原始写法（如 `R01`）
  - `normalizedKey`：统一键，规则为 `R` + 去前导零数字（如 `R01`/`R-001`/`R 1` → `R1`）
- 去重、关联、范围确认一律基于 `normalizedKey`；对用户展示时保留 `rawId`
- 若同一 `normalizedKey` 对应多个不同需求标题，或需求缺少合法 R 编号，必须向用户确认/请求输入 澄清
- `RCA`、`Risk`、`Roadmap` 等非「R+数字」条目不是需求编号，禁止混入需求清单

### 步骤 4: 展示解析结果并确认（→ G4）

⛔ 约束 [GL2]: 不得跳过。展示解析结果及影响评估：

```
[新增] 功能点 1: [标题]
- 影响模块: [模块名]
- 技术层: [UI/数据/状态/模型]
- 文件变动: [新增/修改文件列表]
- 关联需求: [rawId -> normalizedKey]

[修改] 功能点 2: [标题]
- 修改目标: TRD 方案章节 [章节名]
- 修改内容: [简述要改什么]
- 受影响任务: 任务 3、任务 5（已完成，需重置为未完成）
- 波及范围: [其他受影响的 TRD 章节/design 章节]
```

使用 **Codex 用户确认/输入流程** 确认：
- **A：继续** — 确认解析结果，进入下一步自动更新
- **B：调整** — 允许用户修改功能点描述后重新解析
- **C：取消** — 放弃本次操作

### 步骤 5: 自动更新产物（TRD 优先 + 确认循环）

**5.1 更新 TRD（→ G5）**

检索 `seedpacespec/changes/active-change/<变更名>/trds/` 目录下的 TRD 文件。

**新增模式**：
- 在对应技术层章节追加改动描述
- 更新 `## 需求覆盖矩阵`，添加新功能点映射

**修改模式**：
- **就地重写**受影响的 TRD 章节，直接用新方案替换旧方案（默认不保留旧方案；仅当用户明确要求保留时才保留）
- 检查修改是否**波及其他章节**（如改了状态层 → UI 层引用的状态接口需同步改），有波及则一并更新
- 更新 `## 需求覆盖矩阵` 中受影响条目的章节映射

**通用**：
- 在 `## 变更记录` 追加条目，标注操作类型（`[新增]`/`[修改]`）和影响范围

**5.2 用户确认 TRD（必须执行）**

展示更新的 TRD 内容摘要，重点突出新增/修改的技术层改动、涉及的核心文件、与现有方案的兼容性。

> **📋 方案确认 用户可见文案模板**：
> - question: "以上是根据功能点生成的技术方案，请确认："
> - option 1: label="确认正确" description="方案内容正确，继续下一步"
> - option 2: label="需要调整" description="指出需要修改的地方"
> - option 3: label="取消本次添加" description="放弃本次生成的方案内容"

- **选 A（确认正确）**：按顺序执行：①反馈记忆收集（按 `_shared/memory-protocol.md` 执行）→ ②继续步骤 5.3 和 5.4
- **选 B（需要调整）**：收集反馈 → 修正 TRD → 重复步骤 5.1~5.2（循环直到用户确认）
- **选 C（取消）**：回滚 TRD 修改，流程终止

<!-- slot:after_trd_confirmed -->

**5.3 更新 design.md（→ G6，G5 通过后执行）**

- 若功能点涉及新增模块/文件：更新 `## 模块与文件结构（树状图）`，使用 **Codex 用户确认/输入流程** 确认树状图变更
- 若功能点涉及技术决策变更：在 `## 技术决策` 节追加新决策条目
- 更新 `## 功能点、非功能点与约束` 节
- 在 `## 评审记录` 追加变更记录

⛔ 约束 [GL4]: 若步骤 5.2 用户选择「需要调整」或「取消」，则不得执行本步骤。

**5.4 更新 tasks.md（G6 完成后执行）**

**新增模式**：
```markdown
- [ ] [功能点标题] — [简要描述] → 关联需求 [R-XXX]
  - [ ] 子任务 1（对应 TRD §x.x）
  - [ ] 子任务 2（对应 TRD §x.x）
```

**修改模式**：
- **重写**受影响的任务项描述，使其与修改后的 TRD 方案对齐
- **重置已完成但受影响的任务**：若修改的 TRD 章节对应的任务已标记 `[x]`，须改回 `[ ]` 并标注 `⚠️ 方案已变更，需重新实现`
- 不受影响的 `[x]` 任务保持不动

**通用**：
- 需求引用建议写法：`[rawId（normalizedKey）]`，例如 `R01（R1）`
- 任务项须与 design.md 的模块结构、TRD 的技术层改动对齐
- 在文首添加更新备注：「`[日期]` [新增/修改]：[标题]（基于已确认 TRD）」

### 步骤 6: 确认 tasks.md 并询问执行（→ G7）

展示本次新增/更新的任务项清单。

> **📋 任务确认 用户可见文案模板**：
> - question: "以上是本次新增的任务项，是否需要调整？"
> - option 1: label="确认无误" description="任务列表正确，可以保存"
> - option 2: label="需要调整" description="指出需要修改的任务项"
> - option 3: label="暂不执行" description="保存任务但暂时不开始实现"

- **选 A（确认无误）**：按顺序执行：①反馈记忆收集 → ②进入步骤 6.1 询问是否立即执行
- **选 B（需要调整）**：收集反馈 → 修改 tasks.md → 重复步骤 6（循环直到用户确认）
- **选 C（暂不执行）**：保存产物，跳至步骤 7 展示摘要

<!-- slot:after_tasks_confirmed -->

**6.1 询问是否立即执行（仅当用户选择 A 后执行）**

> **📋 立即执行 用户可见文案模板**：
> - question: "是否立即开始实现刚才新增的任务？"
> - option 1: label="立即开始实现" description="现在就开始写代码实现这些任务"
> - option 2: label="稍后再说" description="任务已保存，之后用 $seedpacespec-apply-change 实现"

- **选 A** → 提示用户执行 `$seedpacespec-apply-change <变更名>` 进入实现阶段
- **选 B** → 跳至步骤 7 展示摘要

### 步骤 7: 展示更新摘要

展示本次更新的文件列表及变更概要，运行 `seedpacespec status --change "<变更名>"` 展示最终状态。

**输出格式**：

```

**变更：** <变更名>

| design.md | [新增模块/修改树状图/无变更] | [说明] |
| tasks.md | [追加N个/重写M个/重置K个] | [说明] |
```

## Guardrails

⛔ 以下为全局红线，部分已在执行流程中内联重申。红线编号（GL1、GL2…）与流程中的内联 ⛔ 约束双向索引——读到红线不清楚上下文时，按编号回到对应流程位置查看。

- ⛔ **GL1** 禁止业务代码生成：本工作流**不得**创建、修改或删除业务仓库中的应用源码（`src/` 等）。即使用户明确要求也须拒绝并引导至 `$seedpacespec-apply-change`。允许写入的范围仅限 `seedpacespec/changes/active-change/<name>/` 内产物文档。→ 见 Goal
- ⛔ **GL2** 状态机门禁（G1→G7）是 ASSERTION：任何状态未完成 → 只能向用户确认/请求输入，不得进入后续步骤。凡需要用户交互的步骤，必须停下等用户明确回复。禁止跳过、合并或替用户做决定。→ 见 State Machine Gates
